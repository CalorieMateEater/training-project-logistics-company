# 実装乖離・課題管理簿

## 1. 目的
詳細設計とJava実装の突合で判明した実装漏れ、実装過剰、挙動不一致、設計確認事項を記録する。
詳細設計が曖昧な内容を推測で実装しないよう、課題として管理する。

## 2. 状態定義
| 状態 | 意味 |
| --- | --- |
| `未対応` | 課題を登録したが、対応方針が未確定 |
| `設計確認中` | 詳細設計または基本設計の確認が必要 |
| `実装修正中` | Java実装修正中 |
| `テスト確認中` | 修正済みで、Unit/IT/build確認中 |
| `確認待ち` | 修正とテストが完了し、整合性表の更新待ち |
| `解決` | 設計と実装の再確認が完了 |
| `保留` | 対応時期または採用可否が未定 |

## 4. 区分定義
| 区分 | 意味 |
| --- | --- |
| 実装漏れ | 詳細設計にある処理、項目、テーブル、例外条件、テストがJavaに存在しない |
| 実装過剰 | 詳細設計にない挙動、項目、外部通信、状態更新をJavaが行っている |
| 挙動不一致 | 実装はあるが、条件、値、応答、更新順序、例外扱いが詳細設計と異なる |
| テスト不足 | 実装はあるが、詳細設計上重要な正常系、異常系、境界値、冪等性のテストが不足している |
| 設計確認要 | Java修正可否を判断するための詳細設計が不足または曖昧 |

## 5. 運用ルール
- 課題は削除せず、解決後も履歴として残す。
- `設計確認要`は、推測でJavaを修正しない。
- Java修正を行う場合は、関連する設計IDと確認観点を課題行へ明記する。
- 解決前に `.\gradlew.bat spotlessApply` と `.\gradlew.bat build` の結果を確認する。
- 解決後は[設計実装トレーサビリティ表](./設計実装トレーサビリティ表.md)と[実装整合性確認チェックリスト](./実装整合性確認チェックリスト.md)を更新する。
