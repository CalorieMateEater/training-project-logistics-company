# 矛盾・課題管理簿

## 1. 目的
要件定義と設計の突合で判明した矛盾、要件不足、設計未反映を記録し、推測による下流設計の確定を防止する。

## 2. 状態定義
| 状態 | 意味 |
| --- | --- |
| `未対応` | 課題を登録したが、対応方針が未確定 |
| `方針確認中` | 関係者または要件責任者へ確認中 |
| `設計反映中` | 方針確定後、設計へ反映中 |
| `確認待ち` | 設計反映済みで、整合性再確認待ち |
| `解決` | 要件と設計の再確認が完了 |
| `保留` | 対応時期または採用可否が未定 |

## 4. 運用ルール
- 課題は削除せず、解決後も履歴として残す。
- 要件変更で過去の解決条件が変わった場合は、同じ課題IDを再オープンする。
- 要件不足は詳細設計側だけで推測して解消しない。
- `解決`へ変更する前に、トレーサビリティ表とチェックリストを再確認する。
