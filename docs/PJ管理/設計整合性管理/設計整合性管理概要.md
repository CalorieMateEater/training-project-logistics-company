# 設計整合性管理概要

## 1. 目的
要件定義を正として、基本設計、詳細設計へ要件が漏れなく反映されていることを継続的に確認する。
要件変更後に古い確認結果を流用しないよう、確認サイクルとチェック状態を管理する。

## 2. 管理資料
- [要件トレーサビリティ表](./要件トレーサビリティ表.md)
- [整合性確認チェックリスト](./整合性確認チェックリスト.md)
- [矛盾・課題管理簿](./矛盾・課題管理簿.md)

## 3. 状態定義
| 状態 | 意味 |
| --- | --- |
| 空欄 | 現在の確認サイクルでは未確認 |
| `確認中` | 関連資料を確認中で、結論が確定していない |
| `整合` | 要件と対象設計に明確な矛盾がない |
| `矛盾あり` | 要件と設計の内容が両立せず、設計修正が必要 |
| `要件不足` | 下流設計を確定するための条件、値、データ取得元などが要件側で不足している |
| `対象外` | 当該要件に対する下流設計が不要 |

## 5. 要件更新時のリセット手順
要件の意味、責務、条件、データ項目、IF、非機能条件のいずれかを変更した場合は、下流設計の確認前に次を実施する。

1. 新しい確認サイクルIDを採番する。
2. [要件トレーサビリティ表](./要件トレーサビリティ表.md)の全行について、`状態`、`最終確認日`、`確認者`を空欄へ戻す。
3. [整合性確認チェックリスト](./整合性確認チェックリスト.md)の全行について、`状態`、`最終確認日`、`確認者`を空欄へ戻す。
4. 両資料の確認サイクルIDを新しい値へ更新する。
5. `課題ID`は履歴との接続を維持するため消去せず、変更要件に関係する既存課題を再確認し、必要なら再オープンする。
6. 要件定義、基本設計、詳細設計の順で再確認する。

誤字修正、リンク修正、意味を変えない表記統一だけの場合は、確認状態をリセットしない。

## 6. 確認手順
1. トレーサビリティ表から未確認行を選択する。
2. 要件ID、業務フロー、IF仕様、対外仕様を確認する。
3. 対応する基本設計を確認する。
4. 対応する詳細業務フロー、処理設計、処理機能設計、メソッド設計を確認する。
5. 状態、確認日、確認者を更新する。
6. 矛盾または要件不足があれば課題管理簿へ登録し、課題IDを記載する。
7. 明確な設計矛盾は要件を正として修正し、修正後に再確認する。

## 7. 完了条件
- トレーサビリティ表に空欄または`確認中`がない。
- チェックリストに空欄または`確認中`がない。
- `矛盾あり`が0件である。
- `要件不足`は課題管理簿に登録され、対応方針と担当が明確である。
- 修正した設計書のMarkdown構文、リンク、図表構文に問題がない。

## 8. Codexへの依頼例
```text
設計整合性管理資料の現在の確認サイクルを確認してください。
整合性確認チェックリストの未確認項目から「Bar/Fuga連携」を確認してください。
要件定義を正として、明確な矛盾のみ自社内部設計を修正してください。
要件不足は推測で補完せず、矛盾・課題管理簿へ登録してください。
確認後はトレーサビリティ表とチェックリストを更新してください。
ソースコードは変更しないでください。
```
