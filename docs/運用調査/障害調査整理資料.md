# 障害調査整理資料

## 1. 目的
本書は、障害調査時に原因、影響、確認事項を整理するための観点をまとめる。

個別演習の正解となる具体的なファイル名、行番号、注文ID、エラー値は記載しない。実際の調査では、採取ログと設計書を突合して判断する。

## 2. 調査観点
| 観点 | 確認内容 |
| --- | --- |
| 発生処理 | どのアプリケーション、Job、API、Workerで発生したか |
| 発生時刻 | ログ時刻、アラート時刻、業務イベント時刻が一致するか |
| 対象データ | 対象ファイル、対象リクエストID、対象注文IDを特定できるか |
| 直接原因 | 入力不正、外部API失敗、DB失敗、キュー滞留、業務ルール不一致のどれか |
| 設計突合 | 要件定義、IF仕様、処理設計、業務ルールのどこに対応するか |
| 影響範囲 | 後続処理へ進んだか、途中までDB更新されたか、再実行が必要か |
| 依頼先 | 自社対応か、対向システムへの調査依頼か |

## 3. Foo注文取込で見るべきポイント
- ファイル取込開始ログがあるか
- 対象ファイル名を特定できるか
- 入力検証で止まっているか、外部API呼び出し後に止まっているか
- ログ上のエラーメッセージを連携ファイル設計書の項目仕様へ逆引きできるか
- 自社システム障害か、連携データ不正かを切り分けられるか

## 4. 直受注登録APIで見るべきポイント
- HTTPステータスと例外種別
- リクエストID、顧客ID、商品コードなどの追跡キー
- 入力検証で止まるべき内容が業務ロジックへ到達していないか
- 在庫引当後に失敗した場合、引当解除などの補償が必要か

## 5. 配送会社連携Workerで見るべきポイント
- 対象配送会社
- 営業時間判定結果
- キュー滞留の原因
- 次回送信予定時刻
- 一時対応としてステータス補正または再送抑止が必要か

## 6. 報告文の構成例
```text
対象処理:
対象データ:
確認ログ:
原因:
設計根拠:
影響範囲:
暫定対応:
恒久対応:
追加確認:
```

## 7. 判断時の注意
- ログに出ているエラーメッセージだけで断定せず、入力データと設計書を突合する。
- 途中まで処理が進んでいる場合は、DB更新や外部送信の有無を確認する。
- 外部連携先への依頼が必要な場合は、対象データを特定できる最小限の情報を添える。
