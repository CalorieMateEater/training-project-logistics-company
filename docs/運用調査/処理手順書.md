# 処理手順書

## 1. 目的
本書は、Hoge OrderHubにおける主要処理の起動条件、確認観点、異常時の初動を整理する。

個別障害の正解となる対象ファイル名、対象行、対象値は本書には記載しない。調査時は、実際に配布または採取されたログを確認する。

## 2. Foo注文取込Job
| 項目 | 内容 |
| --- | --- |
| 処理ID | JOB-HOGE-001 |
| 処理名 | FooOrderImportJob |
| 起動契機 | Foo社からHULFTで注文ファイルを都度受信した後、Hoge社のファイル着信領域へ配置されたこと |
| 入力 | Foo社注文ファイル |
| 主な後続処理 | 顧客確認、在庫引当、注文登録、配送会社選定、配送依頼キュー投入 |

### 2.1 正常時の確認観点
- 取込対象ファイルを読み込めていること
- レコード単位の入力検証が完了していること
- 顧客確認API、在庫引当APIへの通信が成功していること
- OrderHubデータストアへ注文が登録されていること
- 配送会社選定結果に応じて、後続処理へ連携されていること

### 2.2 異常時の確認観点
- どの処理で異常終了したか
- 対象ファイル名、対象行、対象注文IDがログから特定できるか
- エラーが入力検証、外部API通信、DB登録、後続キュー投入のどこで発生したか
- 入力検証エラーの場合、連携ファイル設計書の項目仕様と突合できるか
- 外部API通信エラーの場合、送信先、HTTPステータス、タイムアウト、リトライ有無を確認できるか

### 2.3 ログ確認例
以下は確認すべきログ種別の例であり、実際の値は採取ログを参照する。

```text
APP_CONTROLLER_BEFORE method=BatchJobController.importFooOrders(..)
SEND_REQUEST customerStatus customerId=<customer_id>
SEND_REQUEST stockReservation orderId=<order_id> itemCount=<count>
MONITORING_ERROR point=FooOrderImportService.importFile(..) message=<error_message>
APP_CONTROLLER_ERROR method=BatchJobController.importFooOrders(..) message=<error_message>
```

## 3. Shipping Gateway API
| 項目 | 内容 |
| --- | --- |
| 主なAPI | 出荷依頼受付、出荷状態照会、配送結果受信 |
| 主な確認対象 | リクエストID、HTTPステータス、入力検証、顧客確認、在庫引当、配送会社選定 |

### 3.1 異常時の確認観点
- API入口で入力検証に失敗しているか
- 顧客確認または在庫引当後に業務例外が発生していないか
- 既に在庫引当済みの場合、補償処理が必要か
- 同一リクエストIDの重複実行有無

## 4. 配送会社連携Worker
| 項目 | 内容 |
| --- | --- |
| 主な処理 | Bar社またはFuga社への配送依頼送信 |
| 主な確認対象 | 配送会社、営業時間判定、送信状態、次回送信予定時刻、リトライ回数 |

### 4.1 異常時の確認観点
- 配送会社ごとの受付時間内か
- `WAITING_BUSINESS_HOURS` とすべき状態で送信していないか
- タイムアウト、5xx、4xxのどれか
- キュー滞留が営業時間外の待機か、処理失敗による滞留か
