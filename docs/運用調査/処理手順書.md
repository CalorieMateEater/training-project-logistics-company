# 12. 処理手順書

## 1. 文書目的

本書は、`Hoge OrderHub` の `FooOrderImportJob` の実行概要、正常時と異常時のふるまい、および障害調査時に参照すべきログを整理したものである。

## 2. 対象処理

| 項目 | 内容 |
| --- | --- |
| 処理ID | JOB-HOGE-001 |
| 処理名 | FooOrderImportJob |
| 起動タイミング | Foo社ファイル受信都度 |
| 起動契機 | Hoge HULFT受信サーバによる受信完了後、着信S3配置を契機に後続Batchを即時起動 |
| 入力元 | `s3://hoge-orderhub-landing-prod/foo/` |
| 入力ファイル | `FOO_ORDER_20260608011540001_001.dat` |
| 異常終了コード | 8 |

## 3. 正常時手順

1. Hoge HULFT受信サーバが Foo社からの出荷依頼ファイルを受信する
2. 受信ファイルをファイル着信S3に配置する
3. 受信完了イベントにより FooOrderImportJob を起動する
4. Batch が着信S3上の対象ファイルを読み込む
5. 明細レコードのバリデーションを行う
6. Internal API Gateway 経由で Hoge Customer Registryへ顧客確認APIを呼び出す
7. Internal API Gateway 経由で Hoge Stock Keeperへ在庫引当APIを呼び出す
8. 注文元を登録し、Foo予約注文の優先度をもとに優先配送区分を決定する
9. 配送条件判定により Bar社またはFuga社への配送委託先と出荷指示内容を決定する
10. OrderHub データストアへ登録する
11. Batch が配送委託先に応じて `bar-shipment-request-queue.fifo` または `fuga-shipment-request-queue.fifo` へ送信要求を投入する
12. Bar向けの場合、配送会社連携Worker が営業時間判定を行い、平日08:00-18:00のみ Bar社向け出荷指示APIを呼び出す
13. Bar営業時間外の場合、Worker は送信要求を再投入または可視性延長して次回営業時間帯まで待機させる。Fuga向けは営業時間待機を行わずFuga社向け出荷指示APIを呼び出す
14. Bar向け出荷依頼送信成功時のみ、Worker が `billing-plan-queue` へ請求予定メッセージを投入する
15. Bar向け出荷依頼送信成功時のみ、Worker が `order-notice-queue.fifo` へ初回注文通知メッセージを投入する
16. FooOrderImportJob 自体は配送会社向け送信要求の投入完了時点で正常終了する

## 4. 関連する後続処理

### 4.1 配送結果取込

1. Bar社またはFuga社が配送結果をHoge社へ返却する
2. 配送状態取込Workerが配送ステータスを受信する
3. OrderHub データストアへ配送結果を反映する
4. 配送結果返却Workerへ返却通知を依頼する
5. 配送結果返却ファイルを生成し、配送結果返却ファイル出力S3を経由してFoo社へHULFT返却する

### 4.2 日次アーカイブ

1. 日次アーカイブBatchが OrderHub データストアから完了済み注文と配送結果履歴を抽出する
2. 請求連携履歴、通知履歴を含めて売上実績CSVオブジェクトを生成する
3. アーカイブ出力S3から売上アーカイブS3へ保管する

### 4.3 配送連携API

1. Hoge社業務部門の直受注登録クライアントが Hoge社の出荷依頼受付API を呼び出す
2. Hoge社の配送連携APIシステムが OrderHub データストアへ依頼を登録する
3. 配送条件判定後、委託先に応じて `bar-shipment-request-queue.fifo` または `fuga-shipment-request-queue.fifo` へ後続配送処理を連携する
4. Foo社の状態照会クライアント、または Hoge社業務部門の社内参照機能が出荷状態照会APIを呼び出す
5. 配送連携APIシステムが OrderHub データストアを参照して応答する

## 5. 異常時手順

1. Batch がバリデーションエラーを検知する
2. 対象ファイルの処理を異常扱いとする
3. エラー内容をアプリログへ出力する
4. HULFT受信ファイルをエラー領域へ退避する
5. 終了コード8でジョブを終了する
6. バッチ異常ログは CloudWatch Logs と Splunk に転送される

## 8. 調査で注目すべきポイント

- エラーは REST API 呼び出し時ではなく、ファイル読込直後のバリデーションで発生している
- `FOO_ORDER_20260608011540001_001.dat` が対象ファイルである
- ファイル着信先は `s3://hoge-orderhub-landing-prod/foo/` である
- HULFT受信サーバから着信S3へ配置された後に後続Batchが即時起動されているため、アプリが外部ネットワークへ直接接続して受信しているわけではない
- `order_type` と `item_code` の2項目でバリデーションエラーが発生している
- Bar/Fuga、Baz、Qux 向けの後続連携は行われていない
- Hoge社の直受注登録APIや Fuga社配送結果通知API の経路は本障害の直接調査対象ではない
- 入力元システムは `Foo Order Gateway`、担当会社は `Foo販売ソリューション株式会社` である
