# 監視設計書

## 1. 目的
Hoge社受注連携基盤における監視対象、監視方法、閾値、通知先の基本方針を定義する。

## 2. 監視カテゴリ
| カテゴリ | 対象 |
| --- | --- |
| API 監視 | Shipping API、Internal API、Bar/Fuga 結果受付 |
| Batch 監視 | FooOrderImportJob、日次アーカイブBatch |
| Worker 監視 | 配送会社連携Worker、配送状態取込Worker、配送結果返却Worker |
| HULFT 監視 | 受信サーバ、送信サーバ、転送件数 |
| SQS 監視 | キュー滞留、受信失敗 |
| RDS 監視 | CPU、接続数、ストレージ、フェイルオーバー |
| 証明書監視 | 証明書期限 |
| 可用性監視 | 基幹APIの月間稼働率、ALB健全ターゲット数 |

| 対象 | 項目 | 閾値 | 重大度 |
| --- | --- | --- | --- |

## 3.1 可用性SLI/SLO
| 対象 | SLI | SLO | 集計 |
| --- | --- | --- | --- |
| 出荷依頼受付API | 正常応答可能時間 / 月間提供対象時間 | 99.9%以上 | 月次 |
| 出荷状態照会API | 正常応答可能時間 / 月間提供対象時間 | 99.9%以上 | 月次 |
| 配送結果受付API | 正常応答可能時間 / 月間提供対象時間 | 99.9%以上 | 月次 |
| 配送会社連携Worker | 正常稼働時間 / 月間提供対象時間 | 99.9%以上 | 月次 |

- 計画停止を除外する場合は、事前承認済みの保守時間だけを除外対象とする。
- 月間SLO未達またはエラーバジェット消費率100%到達時は、翌月の機能リリースより可用性改善を優先する。
- 単一AZ縮退時もALB健全ターゲット、ECS Desired/Running Task数、RDSフェイルオーバー完了時間を継続監視する。

## 4. 通知方針
| 重大度 | 通知先 | 初動目標 |
| --- | --- | --- |
| Critical | 当番電話、Teams 緊急チャネル | 15分以内 |
| Warning | Teams 運用チャネル | 1時間以内 |
| Info | 日次サマリ | 翌営業日確認 |

## 5. ログ確認観点
- `X-Request-Id`, `X-Correlation-Id` をキーに API ログを追跡する。
- HULFT はファイル ID、ホスト ID、転送時刻で追跡する。
- SQS は送信時の `trace_id` と注文 ID を対応付ける。
- `bar-shipment-request-queue.fifo` は営業時間外滞留と営業時間内滞留を区別して確認する。
- `fuga-shipment-request-queue.fifo` は即時送信前提のため、短時間滞留でも優先度高で確認する。

## 6. ダッシュボード構成
- API 成功率 / 失敗率
- Batch 成功件数 / 失敗件数
- Worker 送受信件数
- SQS 滞留数
- RDS 使用率
- HULFT 送受信件数
