# 監視設計書

## 1. 目的
Hoge社受注連携基盤における監視対象、監視方法、閾値、通知先の基本方針を定義する。

## 2. 監視カテゴリ
| カテゴリ | 対象 |
| --- | --- |
| API 監視 | Shipping API、Internal API、Bar/Fuga 結果受付 |
| Batch 監視 | FooOrderImportJob、日次アーカイブBatch |
| Worker 監視 | 配送会社連携Worker、配送状態取込Worker、配送結果返却Worker |
| HULFT 監視 | 受信サーバ、送信サーバ、転送件数 |
| SQS 監視 | キュー滞留、受信失敗 |
| RDS 監視 | CPU、接続数、ストレージ、フェイルオーバー |
| 証明書監視 | 証明書期限 |

## 3. 監視項目
| 対象 | 項目 | 閾値 |
| --- | --- | --- |
| 出荷依頼受付 API | 5xx 比率 | 5分平均 1% 超 |
| 出荷状態照会 API | P95 レスポンス | 1秒超 |
| 配送会社連携Worker | Bar/Fuga API 失敗件数 | 5分で 5件超 |
| `bar-shipment-request-queue.fifo` | ApproximateAgeOfOldestMessage | 平日営業時間内に 900秒超 |
| `fuga-shipment-request-queue.fifo` | ApproximateAgeOfOldestMessage | 300秒超 |
| 配送状態取込Worker | 受信失敗件数 | 5分で 3件超 |
| FooOrderImportJob | 終了コード | 0 以外で即時通知 |
| HULFT受信 | 未転送着信件数 | 10分以上滞留で通知 |
| HULFT送信 | 未送信返却ファイル件数 | 15分以上滞留で通知 |
| `billing-plan-queue` | ApproximateNumberOfMessagesVisible | 1000 件超 |
| `order-notice-queue.fifo` | ApproximateAgeOfOldestMessage | 600秒超 |
| RDS | FreeStorageSpace | 20% 未満 |
| 証明書 | 有効期限 | 30 日未満 |

## 4. 通知方針
| 重大度 | 通知先 | 初動目標 |
| --- | --- | --- |
| Critical | 当番電話、Teams 緊急チャネル | 15分以内 |
| Warning | Teams 運用チャネル | 1時間以内 |
| Info | 日次サマリ | 翌営業日確認 |

## 5. ログ確認観点
- `X-Request-Id`, `X-Correlation-Id` をキーに API ログを追跡する。
- HULFT はファイル ID、ホスト ID、転送時刻で追跡する。
- SQS は送信時の `trace_id` と注文 ID を対応付ける。
- `bar-shipment-request-queue.fifo` は営業時間外滞留と営業時間内滞留を区別して確認する。
- `fuga-shipment-request-queue.fifo` は即時送信前提のため、短時間滞留でも優先度高で確認する。

## 6. ダッシュボード構成
- API 成功率 / 失敗率
- Batch 成功件数 / 失敗件数
- Worker 送受信件数
- SQS 滞留数
- RDS 使用率
- HULFT 送受信件数
