# 障害対応手順書

## 1. 目的
運用当番が障害発生時に初動、切り分け、暫定復旧、エスカレーションを実施するための基本手順を定義する。

## 2. 初動共通手順
1. 監視通知または問い合わせ内容を記録する。
2. 発生時刻、対象 IF、対象注文 ID、対象ファイル名を特定する。
3. 影響範囲を `単一注文 / 単一連携 / 全体障害` で分類する。
4. 直近の変更有無を確認する。
5. 必要に応じて当番、開発、基盤担当へエスカレーションする。

## 3. 障害パターン別対応

### 3.1 Foo 注文ファイル未取込
- 確認順:
  1. HULFT受信サーバの着信有無
  2. `landing` S3 への転送有無
  3. FooOrderImportJob の起動履歴
  4. バリデーションエラー有無
- 暫定対応:
  - S3 着信済みなら Batch 再実行
  - HULFT 着信止まりなら HULFT サーバ調査

### 3.2 配送会社向け出荷依頼失敗
- 確認順:
  1. 配送会社連携Worker エラーログ
  2. Bar/Fuga API 応答コード
  3. Bar社向けの場合、発生時刻が平日08:00-18:00内か
  4. `t_shipment_request.shipment_request_status` と `next_request_after`
  5. VPN 疎通
  6. 冪等キー重複有無
- 暫定対応:
  - タイムアウトなら再送判定
  - Bar社営業時間外で `PENDING` のまま再送している場合は、`WAITING_BUSINESS_HOURS` へ待機させ、翌営業日08:00以降に再送確認
  - 4xx 業務エラーなら業務担当確認

### 3.3 配送会社結果通知未反映
- 確認順:
  1. Bar/Fuga 結果受付 API 着信ログ
  2. 配送状態取込Worker ログ
  3. 配送会社別の`status_seq`重複・旧イベント判定
  4. OrderHub データ更新有無
- 暫定対応:
  - 受信済み未反映なら Worker 再処理

### 3.4 Foo 向け返却未送信
- 確認順:
  1. 配送結果返却Worker の返却ファイル生成有無
  2. `status` S3 出力有無
  3. HULFT送信サーバの取得結果
  4. HULFT 送信ログ
- 暫定対応:
  - S3 出力済みなら送信ジョブ再実行

### 3.5 Foo 状態照会 API / Hoge社内参照応答不良
- 確認順:
  1. 出荷状態照会 API の 4xx / 5xx ログ
  2. Private ALB の Target HealthyHostCount
  3. API キー認証エラー有無
  4. Foo社 VPN または Hoge社内接続経路の疎通
  5. OrderHub データ参照遅延有無
- 暫定対応:
  - 認証エラーなら接続元設定と API キーを確認
  - 5xx またはタイムアウトなら API タスク再配置を検討
  - 特定社のみ影響なら会社別 VPN / 接続元制御を優先確認

### 3.6 Hoge直受注登録API 5xx
- 確認順:
  1. `partnerRequestId`、`X-Trace-Id`、`X-Client-System-Id`
  2. `MONITORING_API_UNEXPECTED` または `MONITORING_API_VALIDATION` の有無
  3. リクエストボディの `deliveryConstraint.temperatureZone`、`shipmentPreference`
  4. 顧客確認・在庫引当が実行済みか
  5. 在庫引当済みの場合、補償解除の要否
- 暫定対応:
  - 入力不足が原因なら、呼出元にリクエスト修正を依頼
  - 5xxで在庫引当済みの場合は、該当注文の引当状態を確認し、必要に応じて引当解除

### 3.7 Bar営業時間外のキュー滞留
- 確認順:
  1. アラート発生時刻が Bar社営業時間外か
  2. `bar-shipment-request-queue.fifo` の最古メッセージ滞留時間
  3. `ShipmentDispatchWorker` の `APP_BATCH_RECORD_WAIT reason=outsideBusinessHours` ログ有無
  4. `MONITORING_BATCH_ERROR` が5分間隔で繰り返されていないか
  5. `shipment_request_status` が `WAITING_BUSINESS_HOURS` か `PENDING` か
- 暫定対応:
  - 営業時間外待機であれば、営業時間内に再送されることを確認
  - 営業時間外にもBar APIへ送信している場合は、対象出荷依頼を `WAITING_BUSINESS_HOURS` に補正し、再送時刻を翌営業日08:00以降にする

## 4. エスカレーション基準
| 条件 | エスカレーション先 |
| --- | --- |
| 全注文影響 | 運用責任者、開発責任者 |
| 30分超未復旧 | 基盤担当、開発担当 |
| データ欠損疑い | 開発責任者、DBA |
| 外部企業側障害疑い | 対外連携窓口 |
