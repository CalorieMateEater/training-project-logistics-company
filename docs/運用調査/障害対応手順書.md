# 障害対応手順書

## 1. 目的
運用当番が障害発生時に初動、切り分け、暫定復旧、エスカレーションを実施するための基本手順を定義する。

## 2. 初動共通手順
1. 監視通知または問い合わせ内容を記録する。
2. 発生時刻、対象 IF、対象注文 ID、対象ファイル名を特定する。
3. 影響範囲を `単一注文 / 単一連携 / 全体障害` で分類する。
4. 直近の変更有無を確認する。
5. 必要に応じて当番、開発、基盤担当へエスカレーションする。

## 3. 障害パターン別対応

### 3.1 Foo 注文ファイル未取込
- 確認順:
  1. HULFT受信サーバの着信有無
  2. `landing` S3 への転送有無
  3. FooOrderImportJob の起動履歴
  4. バリデーションエラー有無
- 暫定対応:
  - S3 着信済みなら Batch 再実行
  - HULFT 着信止まりなら HULFT サーバ調査

### 3.2 Bar 向け出荷依頼失敗
- 確認順:
  1. 配送会社連携Worker エラーログ
  2. Bar API 応答コード
  3. VPN 疎通
  4. 冪等キー重複有無
- 暫定対応:
  - タイムアウトなら再送判定
  - 4xx 業務エラーなら業務担当確認

### 3.3 Bar 結果通知未反映
- 確認順:
  1. Bar 結果受付 API 着信ログ
  2. 配送状態取込Worker ログ
  3. `status_seq` 逆転拒否有無
  4. OrderHub データ更新有無
- 暫定対応:
  - 受信済み未反映なら Worker 再処理

### 3.4 Foo 向け返却未送信
- 確認順:
  1. 配送結果返却Worker の返却ファイル生成有無
  2. `status` S3 出力有無
  3. HULFT送信サーバの取得結果
  4. HULFT 送信ログ
- 暫定対応:
  - S3 出力済みなら送信ジョブ再実行

### 3.5 Foo / Fuga 状態照会 API 応答不良
- 確認順:
  1. 出荷状態照会 API の 4xx / 5xx ログ
  2. Private ALB の Target HealthyHostCount
  3. API キー認証エラー有無
  4. Foo社 / Fuga社 VPN 疎通
  5. OrderHub データ参照遅延有無
- 暫定対応:
  - 認証エラーなら接続元設定と API キーを確認
  - 5xx またはタイムアウトなら API タスク再配置を検討
  - 特定社のみ影響なら会社別 VPN / 接続元制御を優先確認

## 4. エスカレーション基準
| 条件 | エスカレーション先 |
| --- | --- |
| 全注文影響 | 運用責任者、開発責任者 |
| 30分超未復旧 | 基盤担当、開発担当 |
| データ欠損疑い | 開発責任者、DBA |
| 外部企業側障害疑い | 対外連携窓口 |
