# 03. 外部インターフェース一覧

## 1. 文書目的

本書は、Hogeシステムと関係する主要外部インターフェースを一覧化し、送受信方向、方式、同期区分、および調査時の着眼点を整理する。

## 3. 調査時の見方

### 3.1 主入力IF

`IF-FOO-HOGE-001` は、`FooOrderImportJob` が読み込む出荷依頼ファイルの入力元である。  
ファイル起点の調査では、最初に確認する対象IFとなる。

### 3.2 関連IF

- `IF-HOGE-BAR-001` は Hoge社が配送条件判定後に送信待ちキューを経由して Bar社へ配送指示を連携する後続APIである
- `IF-HOGE-FUGA-001` は Hoge社が特殊配送条件に合致した案件を Fuga社へ配送指示する後続APIである
- `IF-HOGE-BAR-001` の送信電文には、注文元コードと優先配送区分が含まれる
- `IF-HOGE-FUGA-001` の送信電文には、倉庫場所、温度帯、サイズ区分などの特殊配送判定結果が含まれる
- `IF-HOGE-STK-001` は在庫引当だけでなく、倉庫場所コードを確定して OrderHub へ返す社内連携である
- `IF-HOGE-STK-002` は出荷前取消や障害復旧時に引当済在庫を解放するための社内連携である
- `IF-HOGE-STK-003` は `配送受付済` 受信時に引当済在庫を出荷確定へ更新するための社内連携である
- `IF-BAR-HOGE-002` は Bar社が標準配送結果をHoge社へ返却する後続連携である
- `IF-FUGA-HOGE-002` は Fuga社が特殊配送結果をHoge社へ返却する後続連携である
- `IF-HOGE-FOO-001` は Hoge社が注文受付完了を Foo社へ通知するためのファイル連携であり、Bar社受付とは独立して返却される
- `IF-FOO-HOGE-003` は Foo社の状態照会クライアントがHoge社APIを呼び出す同期連携である
- 状態照会APIは現行要件では Foo社の照会を主対象とし、`partner_order_id` をキーに最新状態を返却する
- `IF-HOGE-FOO-002` は Hoge社が配送結果をFoo社へ返却するためのファイル連携である
- `IF-HOGE-BAZ-001` と `IF-HOGE-QUX-001` は Hoge社クラウド基盤上の中間キューを介した外部クラウド向け非同期連携である
- `IF-HOGE-ARC-001` は `日次アーカイブBatch` が完了済みデータを `S3` に保管する内部連携である
- Foo社は `HULFTで出荷依頼`、`APIで状態照会`、`HULFTで結果受領` を行う
- Fuga社は `特殊配送の受託` と `配送結果通知` を行う
- Hoge社は受注時に注文元を保持し、Foo社の予約注文では優先度をもとに配送優先区分を決定し、倉庫場所・温度帯・サイズ区分・配送先条件で Bar/Fuga を選定する
- Hoge社は24時間365日で依頼受付するが、Bar社向け送信は平日08:00-18:00の営業時間に合わせて行う

## 4. 補足

障害調査では、方式だけでなく以下を組み合わせて判断すること。

- ファイル名
- ファイル着信先
- 送受信方向
- 中継サーバと入口装置
- API呼出主体
- 処理フロー
- システム一覧に記載された担当会社
