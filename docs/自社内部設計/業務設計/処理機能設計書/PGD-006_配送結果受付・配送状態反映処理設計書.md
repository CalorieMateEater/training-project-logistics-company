# PGD-006 配送結果受付・配送状態反映処理設計書

## 1. 基本情報
| 項目 | 内容 |
| --- | --- |
| 処理機能ID | `PGD-006` |
| 関連処理設計書ID | `PDS-004`, `PDS-005` |
| 論理機能名 | 配送結果受付・配送状態反映 |
| 概要 | Bar社またはFuga社から配送結果を受け付けて状態反映要求へ引き継ぎ、非同期処理で履歴登録、最新状態更新、Bar社配送受付済時の在庫出荷確定、後続通知送信までを行う。 |

## 2. 入出力
| 区分 | 内容 |
| --- | --- |
| 入力 | 配送結果受付APIリクエスト |
| 出力 | 配送結果受付応答、配送状態履歴、最新配送状態、後続通知メッセージ、後続通知履歴 |

## 3. 主処理
1. 配送結果受付APIが対象注文、配送会社受付番号、状態連番を確認する。
2. 同一内容再送は状態反映要求を作成せず受付済として終了する。旧着は状態反映要求を作成せず、Bar社には受付済、Fuga社には競合応答とする。内容相違と遷移不正は配送会社を問わず競合応答とする。
3. 正常な新規イベントは受付監査を記録し、配送状態取込Worker向け状態反映要求へ引き継ぐ。
4. 配送状態取込WorkerがBar社の初回 `配送受付済` 反映時のみ在庫出荷確定を実行する。
5. 配送状態履歴を登録し、最新配送状態と注文状態を更新する。
6. Foo注文ではFoo返却通知とQux状態更新通知を登録し、配送完了時は注文元を問わずBaz確定請求通知を各SQSキューへ投入する。
7. Baz/Quxキュー投入結果に応じて通知履歴を `SENT` または `ERROR` へ更新する。

## 4. 制御方針
- 重複状態は受付済としてスキップし、再反映しない。旧着も再反映せず、Bar社には `202`、Fuga社には `409` を返却する。
- Bar社の `配送受付済` の初回反映時のみ、在庫引当結果を出荷確定へ更新する。
- Foo注文では配送結果返却通知を登録する。
- 配送完了時は請求確定通知を追加登録する。
- Qux通知はFoo注文だけを対象とし、Hoge直受注では生成しない。
- 配送結果受付APIは業務状態を直接更新せず、配送状態取込Workerへ引き継いだ時点で `202 Accepted` を返却する。
- Baz/Qux向け通知失敗では配送状態更新をロールバックせず、通知履歴を `ERROR` として通知だけを再送する。
