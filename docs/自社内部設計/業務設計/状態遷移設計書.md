# 状態遷移設計書

## 1. 目的
OrderHub 内で管理する注文状態、配送依頼状態、配送状態、通知状態の遷移条件を整理する。

## 2. 対象状態
- 注文状態
- 配送依頼状態
- 配送状態
- 通知状態

## 3. 注文状態遷移
| 現在状態 | イベント | 次状態 | 更新主体 |
| --- | --- | --- | --- |
| `RECEIVED` | 受付完了かつ `shipping_release_at` 未指定または到来済 | `WAITING_BAR_REQUEST` | FooOrderImportJob / 出荷依頼受付API |
| `RECEIVED` | 受付完了かつ `shipping_release_at` 未到来 | `WAITING_SHIPPING_RELEASE` | FooOrderImportJob / 出荷依頼受付API |
| `WAITING_SHIPPING_RELEASE` | 解放日時到来かつ Bar社営業時間外 | `WAITING_BAR_REQUEST` | 配送会社連携Worker |
| `WAITING_SHIPPING_RELEASE` | 解放日時到来かつ Bar社送信成功 | `BAR_ACCEPTED` | 配送会社連携Worker |
| `WAITING_BAR_REQUEST` | Bar社出荷依頼送信成功 | `BAR_ACCEPTED` | 配送会社連携Worker |
| `BAR_ACCEPTED` | 配送準備中受信 | `PREPARING_FOR_SHIPMENT` | 配送状態取込Worker |
| `PREPARING_FOR_SHIPMENT` | 配送中受信 | `IN_DELIVERY_FLOW` | 配送状態取込Worker |
| `IN_DELIVERY_FLOW` | 配送完了受信 | `COMPLETED` | 配送状態取込Worker |
| `IN_DELIVERY_FLOW` | 配送失敗 / 持戻り / 住所不備受信 | `EXCEPTION` | 配送状態取込Worker |
| `EXCEPTION` | 再配達待ち受信 | `REDISPATCH_PENDING` | 配送状態取込Worker |
| `REDISPATCH_PENDING` | 配送完了受信 | `COMPLETED` | 配送状態取込Worker |
| `BAR_ACCEPTED` | キャンセル受信 | `CANCELLED` | 配送状態取込Worker |

## 4. 配送依頼状態遷移
| 現在状態 | イベント | 次状態 |
| --- | --- | --- |
| `PENDING` | `shipping_release_at` 未到来 | `PENDING` |
| `PENDING` | Bar社営業時間外判定 | `WAITING_BUSINESS_HOURS` |
| `WAITING_BUSINESS_HOURS` | 次回Bar社営業時間到来 | `PENDING` |
| `PENDING` | Bar社 2xx 応答 | `ACCEPTED` |
| `WAITING_BUSINESS_HOURS` | Bar社 2xx 応答 | `ACCEPTED` |
| `PENDING` | 4xx / 5xx / タイムアウトで再実行上限超過 | `FAILED` |
| `WAITING_BUSINESS_HOURS` | 4xx / 5xx / タイムアウトで再実行上限超過 | `FAILED` |
| `FAILED` | 運用再実行 | `PENDING` |

## 5. 配送状態遷移
| 状態コード | 状態名 | 遷移先候補 |
| --- | --- | --- |
| `ACCEPTED` | 配送受付済 | `PREPARING`, `CANCELLED` |
| `PREPARING` | 配送準備中 | `IN_TRANSIT`, `CANCELLED` |
| `IN_TRANSIT` | 配送中 | `DELIVERED`, `DELIVERY_FAILED`, `RETURNED_TO_BASE`, `ADDRESS_ERROR`, `REDISPATCH_PENDING` |
| `DELIVERED` | 配送完了 | 終端 |
| `DELIVERY_FAILED` | 配送失敗 | `REDISPATCH_PENDING`, 終端 |
| `RETURNED_TO_BASE` | 持戻り | `REDISPATCH_PENDING`, 終端 |
| `ADDRESS_ERROR` | 住所不備 | `REDISPATCH_PENDING`, 終端 |
| `REDISPATCH_PENDING` | 再配達待ち | `IN_TRANSIT`, `DELIVERED`, `DELIVERY_FAILED` |
| `CANCELLED` | キャンセル | 終端 |

## 6. 通知状態遷移
### 6.1 Baz社 / Qux社向け通知
| 現在状態 | イベント | 次状態 |
| --- | --- | --- |
| `PENDING` | メッセージ投入成功 | `SENT` |
| `PENDING` | キュー投入失敗 | `ERROR` |
| `ERROR` | 手動再送 | `PENDING` |

### 6.2 Foo社向け配送状態返却
| 現在状態 | イベント | 次状態 |
| --- | --- | --- |
| `PENDING` | 返却ファイル生成およびHULFT送信成功 | `SENT` |
| `PENDING` | 返却ファイル生成またはHULFT送信失敗 | `ERROR` |
| `ERROR` | 再送実施 | `PENDING` |

## 7. 制御ルール
- 配送状態取込Worker は `status_seq` が現在保持値より大きい場合のみ最新状態を更新する。
- 終端状態 `COMPLETED`, `CANCELLED` に達した注文は、再度 `WAITING_BAR_REQUEST` に戻さない。
- Bar社営業時間外は、配送会社連携Worker は配送依頼を失敗扱いにせず `WAITING_BUSINESS_HOURS` を維持する。
- 配送会社連携Worker の Bar社送信中状態は実装上の短時間処理として扱い、永続化状態としては `PENDING` / `WAITING_BUSINESS_HOURS` / `ACCEPTED` / `FAILED` を採用する。
