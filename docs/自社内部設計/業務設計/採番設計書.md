# 採番設計書

## 1. 目的
本書は、Hoge社の出荷・配送管理で採番する業務ID、履歴ID、通知ID、相関IDの形式、採番契機、重複防止方針を定義する。

## 2. 採番方針
- 業務IDは人手調査しやすいように接頭辞を持つ。
- 採番タイミングは「業務的に同一性が確定する時点」とする。
- 外部リクエスト再送時は、同一業務とみなすIDを再利用し、新規業務の場合のみ新規採番する。

## 3. 採番一覧
| ID種別 | 項目名 | 形式 | 採番契機 | 一意性範囲 |
| --- | --- | --- | --- | --- |
| 注文ID | `order_id` | `ORD-YYYYMMDD-XXXXXXXX` | 注文受付確定時 | 全社・全期間 |
| 出荷依頼ID | `shipment_request_id` | `SHP-YYYYMMDD-XXXXXXXX` | Bar向け出荷依頼生成時 | 全社・全期間 |
| 通知ID | `notification_id` | `NTF-YYYYMMDDHHMMSS-XXXXXX` | 通知要求生成時 | 全社・全期間 |
| 連携履歴ID | `interface_history_id` | `IFH-YYYYMMDDHHMMSS-XXXXXX` | IF送受信記録作成時 | 全社・全期間 |
| アーカイブ実行ID | `archive_execution_id` | `ARC-YYYYMMDD-NN` | 日次アーカイブ開始時 | 実行日単位 |
| 相関ID | `correlation_id` | `COR-YYYYMMDDHHMMSS-XXXXXX` | 外部要求受信時 | 全社・全期間 |

## 4. IDごとの詳細

### 4.1 注文ID
| 項目 | 内容 |
| --- | --- |
| 例 | `ORD-20260616-00018452` |
| 採番主体 | FooOrderImportJob、出荷依頼受付API |
| 再利用条件 | 同一 `partner_order_id` の重複受付では再採番しない |
| 備考 | 注文ヘッダ、注文明細、配送状態、通知履歴の親IDとして利用する |

### 4.2 出荷依頼ID
| 項目 | 内容 |
| --- | --- |
| 例 | `SHP-20260616-00003128` |
| 採番主体 | FooOrderImportJob、出荷依頼受付API |
| 再利用条件 | 同一注文に対する同一配送依頼再送では再採番しない |
| 備考 | Bar社向け送信単位。冪等送信の軸になる |

### 4.3 通知ID
| 項目 | 内容 |
| --- | --- |
| 例 | `NTF-20260616142015-000421` |
| 採番主体 | 注文受付通知Worker、配送状態取込Worker、配送会社連携Worker |
| 再利用条件 | 同一通知種別・同一イベントの再送では既存IDを再利用する |
| 備考 | 通知履歴管理、再送管理に利用する |

### 4.4 連携履歴ID
| 項目 | 内容 |
| --- | --- |
| 例 | `IFH-20260616142103-000778` |
| 採番主体 | すべてのIF送受信処理 |
| 再利用条件 | なし |
| 備考 | 送信と受信を相関IDで紐づけるため、連携履歴自体は毎回採番する |

### 4.5 アーカイブ実行ID
| 項目 | 内容 |
| --- | --- |
| 例 | `ARC-20260616-01` |
| 採番主体 | 日次アーカイブBatch |
| 再利用条件 | 同一未完了実行の再開時は同一IDを継続利用する |
| 備考 | 1日複数回実行に備え末尾シーケンスを持つ |

### 4.6 相関ID
| 項目 | 内容 |
| --- | --- |
| 例 | `COR-20260616142210-000993` |
| 採番主体 | HULFT受信トリガ処理、外部API受信処理 |
| 再利用条件 | 同一リクエスト再試行では呼出元相関IDを優先し、なければ新規採番する |
| 備考 | CloudWatch Logs、Splunk、連携履歴で共通利用する |

## 5. 外部要求IDとの関係
| 外部識別子 | 利用元 | 役割 | Hoge側の扱い |
| --- | --- | --- | --- |
| `partner_order_id` | Foo社 / Fuga社 | 対向先注文識別子 | 業務上の照会キーとして保持 |
| `partner_request_id` | Fuga社 | API受付冪等判定キー | 重複受付検知に利用 |
| `bar_shipment_id` | Bar社 | 配送依頼・配送結果関連付けキー | 配送結果取込時に出荷依頼へ紐づける |

## 6. 採番方式
- 物理採番は PostgreSQL シーケンスを利用する。
- 日付部は採番時点の JST を利用する。
- 可読性を保つため、シーケンス部は固定長ゼロ埋めとする。
- 将来の並列処理増加を考慮し、シーケンスはGaplessを要求しない。

## 7. 障害時方針
- 採番後にロールバックした番号は再利用しない。
- 再送時は「同一業務であることを示す外部識別子」がある場合のみ同一ID再利用可とする。
- ID重複検知時は採番異常として `SYSTEM_ERROR` 扱いとし、即時調査対象とする。
