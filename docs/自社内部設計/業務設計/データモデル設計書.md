# データモデル設計書

## 1. 目的
Hoge OrderHub が保持する業務データの主なエンティティ、主キー、関係、更新責務を整理する。

## 2. 設計方針
- 業務上の正本データは OrderHub データストアで管理する。
- 外部連携の送受信記録は、業務データとは分離した連携履歴エンティティで保持する。
- 配送結果は最新状態と履歴を分けて保持し、障害調査と監査追跡を両立する。
- 冪等制御に必要なキーは業務トランザクションと同一データストアで管理する。

## 3. エンティティ一覧
| エンティティ | 概要 | 主キー |
| --- | --- | --- |
| 注文ヘッダ | 受注の代表情報 | `order_id` |
| 注文明細 | 受注商品の明細 | `order_id + order_line_no` |
| 出荷依頼 | 配送会社への出荷依頼単位 | `shipment_request_id` |
| 配送状態最新 | 注文ごとの現在配送状態 | `order_id` |
| 配送状態履歴 | Bar社から受信した状態変化履歴 | `order_id + status_seq` |
| 顧客確認結果 | 顧客確認 API 呼出結果 | `order_id` |
| 在庫引当結果 | 在庫引当 API 呼出結果 | `order_id + order_line_no` |
| 通知履歴 | Baz社、Qux社、Foo社返却の通知記録 | `notification_id` |
| 連携履歴 | 外部 IF 単位の送受信記録 | `interface_history_id` |
| 冪等受付履歴 | Bar社向け送信時の再送制御記録 | `idempotency_key` |
| アーカイブ管理 | アーカイブ出力単位の管理情報 | `archive_execution_id` |

## 4. 主要エンティティ

### 4.1 注文ヘッダ
| 項目 | 概要 |
| --- | --- |
| `order_id` | OrderHub 内部注文 ID |
| `partner_order_id` | 提携先側注文 ID |
| `partner_request_id` | Fuga社API要求ID。Fuga照会キーとしても利用 |
| `order_source` | 受注元種別。`FOO`, `FUGA` |
| `partner_priority_level` | 提携先から受けた優先度。Foo注文のみ保持 |
| `shipping_priority_class` | Bar向け送信時の優先配送区分。`NORMAL`, `PRIORITY` |
| `customer_id` | 顧客 ID |
| `order_status` | 受注状態 |
| `shipment_status` | 配送状態 |
| `carrier_code` | 選定された配送会社コード |
| `shipping_release_at` | 予約出荷の解放日時 |
| `created_at` | 注文登録日時 |
| `updated_at` | 最終更新日時 |

### 4.2 出荷依頼
| 項目 | 概要 |
| --- | --- |
| `shipment_request_id` | Bar社向け出荷依頼 ID |
| `order_id` | 対象注文 ID |
| `carrier_code` | `BAR` |
| `order_source` | `FOO`, `FUGA` |
| `partner_priority_level` | 提携先優先度のスナップショット |
| `shipping_priority_class` | Bar向け優先配送区分 |
| `routing_rule_id` | 選定ルール識別子 |
| `shipment_request_status` | 依頼状態 |
| `queue_enqueued_at` | Bar送信待ちキュー登録日時 |
| `next_request_after` | 次回Bar送信予定日時 |
| `requested_at` | 出荷依頼送信日時 |
| `bar_accepted_at` | Bar社受付応答受信日時 |
| `bar_accept_no` | Bar社受付番号 |

### 4.3 配送状態最新
| 項目 | 概要 |
| --- | --- |
| `order_id` | 対象注文 ID |
| `latest_status_code` | 最新配送状態 |
| `latest_status_name` | 最新配送状態名 |
| `latest_display_status_name` | 外部表示用に正規化した状態名 |
| `latest_status_seq` | 最新受理済みシーケンス |
| `latest_reason_category` | 理由分類コード |
| `address_corrected` | 住所補正有無 |
| `address_correction_level` | 住所補正レベル |
| `latest_status_at` | 最新状態日時 |
| `last_received_at` | 最終受信日時 |

### 4.4 配送状態履歴
| 項目 | 概要 |
| --- | --- |
| `order_id` | 対象注文 ID |
| `status_seq` | Bar社通知順序 |
| `status_code` | 配送状態コード |
| `status_name` | 配送状態名 |
| `event_occurred_at` | Bar社側イベント発生日時 |
| `received_at` | Hoge社受信日時 |
| `raw_payload_hash` | 受信ペイロード検証用ハッシュ |
| `reason_category` | 外部表示用理由分類 |
| `address_corrected` | 住所補正有無 |
| `address_correction_level` | 住所補正レベル |

## 5. エンティティ関連
```text
注文ヘッダ 1 --- n 注文明細
注文ヘッダ 1 --- 0..1 出荷依頼
注文ヘッダ 1 --- 1 配送状態最新
注文ヘッダ 1 --- n 配送状態履歴
注文ヘッダ 1 --- n 通知履歴
注文ヘッダ 1 --- n 連携履歴
出荷依頼 1 --- 1 冪等受付履歴
```

## 6. 更新責務
| エンティティ | 更新主体 |
| --- | --- |
| 注文ヘッダ | FooOrderImportJob、出荷依頼受付API、配送状態取込Worker |
| 注文明細 | FooOrderImportJob、出荷依頼受付API |
| 出荷依頼 | FooOrderImportJob、出荷依頼受付API、配送会社連携Worker |
| 配送状態最新 | 配送状態取込Worker |
| 配送状態履歴 | 配送状態取込Worker |
| 顧客確認結果 | FooOrderImportJob、出荷依頼受付API |
| 在庫引当結果 | FooOrderImportJob、出荷依頼受付API |
| 通知履歴 | 配送会社連携Worker、配送結果返却Worker |
| 連携履歴 | 全 IF 呼出処理 |
| 冪等受付履歴 | 配送会社連携Worker |
| アーカイブ管理 | 日次アーカイブBatch |

## 7. 索引方針
- 注文ヘッダ: `partner_order_id`, `order_status`, `shipment_status`, `created_at`
- 出荷依頼: `shipment_request_id`, `bar_accept_no`, `shipment_request_status`, `shipping_priority_class`
- 配送状態履歴: `order_id`, `status_seq`, `event_occurred_at`
- 連携履歴: `if_id`, `request_datetime`, `result_code`

## 8. 保持期間
| データ | 保持期間 |
| --- | --- |
| 注文ヘッダ、注文明細 | 3年 |
| 配送状態履歴 | 3年 |
| 通知履歴 | 1年 |
| 連携履歴 | 1年 |
| 冪等受付履歴 | 90日 |
