# セキュリティ設計書

## 1. 目的
Hoge社受注連携基盤における IAM、認証、暗号化、秘密情報、監査、防御、脆弱性対応、証明書管理の方針を定義する。

## 2. IAM / 認証
| 観点 | 方針 |
| --- | --- |
| 人の認証 | IAM Identity Center |
| MFA | 管理者、運用者必須 |
| Root ユーザー | 日常利用禁止 |
| アプリ認証 | IAM Role 利用、長期キー禁止 |
| 社外 API 認証 | Foo/Fuga連携は mTLS + APIキーを基本とする。Bar連携は対向合意に従いAPIキー + 接続元IP制限とする |
| 社内 API 認証 | mTLS または OAuth2 Client Credentials |
| 外部SQS Consumer認証 | Baz/Qux社AWSアカウントの専用IAM Roleによるクロスアカウント認証 |

## 3. 最小権限
- ECS タスクロールはコンポーネント単位で分離する。
- S3 はバケット単位、SQS はキュー単位で権限を限定する。
- KMS は利用ロールを CMK ごとに限定する。

## 4. 暗号化
| 対象 | 方針 |
| --- | --- |
| S3 | SSE-KMS |
| RDS | KMS による保管時暗号化 |
| EBS | KMS による保管時暗号化 |
| SQS | SSE-KMS |
| Secrets Manager | KMS |
| 通信 | TLS 1.2 以上 |

## 5. 秘密情報管理
| 対象 | 保管先 |
| --- | --- |
| DB 接続情報 | Secrets Manager |
| API キー | Secrets Manager |
| クライアント証明書 | Secrets Manager + ACM Private CA 管理台帳 |
| アプリ設定値 | SSM Parameter Store |

## 6. 監査
| 項目 | 方針 |
| --- | --- |
| CloudTrail | 全リージョン、全アカウント集約 |
| AWS Config | 有効化 |
| GuardDuty | 有効化 |
| Security Hub | 有効化 |
| 監査ログ保管期間 | 7年 |

## 7. 境界防御
| 項目 | 方針 |
| --- | --- |
| VPN | 対外接続は閉域 VPN のみ |
| Security Group | 最小権限 |
| NACL | サブネット境界制御 |
| WAF | 非公開閉域 API のため未採用 |
| Shield Advanced | 未採用 |
| Network Firewall | 今回は未採用 |

## 8. 脆弱性対応
| 項目 | 方針 |
| --- | --- |
| ECR スキャン | Push 時自動実行 |
| Inspector | ECS/EC2 対象に有効化 |
| OS 更新 | 月次、緊急時随時 |
| コンテナ更新 | 月次ベースイメージ更新 |

## 9. 証明書管理
| 項目 | 方針 |
| --- | --- |
| サーバ証明書 | ACM Private CA |
| クライアント証明書 | 社内 CA 発行 |
| 失効管理 | CRL 日次同期 |
| 更新手順 | 満了 30 日前から更新開始 |

## 10. 補足
- インターネット公開を採用しないため、WAF は必須要件としない。
- 社外APIの認証方式は共通方針より対向別IF基盤設計を優先し、Bar連携では共有APIキー、接続元IP制限、VPN閉域網、TLS 1.2以上を組み合わせる。
- Baz/QuxのSQSアクセスは各社VPCのSQS Interface VPC Endpointを利用し、パブリックインターネット経由のアクセスを許可しない。
- ただし閉域接続先追加や公開要件追加時は、防御方針を再評価する。
