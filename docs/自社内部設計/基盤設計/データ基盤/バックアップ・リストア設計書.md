# バックアップ・リストア設計書

## 1. 目的
OrderHub の RDS、S3、Secrets を対象に、バックアップ取得方針とリストア手順の基本方針を整理する。

## 2. 対象一覧
| 対象 | バックアップ方式 | 目標 |
| --- | --- | --- |
| RDS PostgreSQL | 自動バックアップ + PITR + 手動スナップショット | RTO 30分、RPO 5分 |
| S3 バケット | Versioning + Lifecycle | 誤削除から復旧可能 |
| Secrets Manager | バージョン管理 | 誤更新から復旧可能 |
| HULFT サーバ構成 | AMI + 起動テンプレート | 60分以内再構築 |

## 3. RDS バックアップ
| 項目 | 設定 |
| --- | --- |
| 自動バックアップ保持 | 35日 |
| PITR | 有効 |
| 手動スナップショット | 月次 12世代 |
| 取得時間帯 | 04:30-05:00 |

## 4. S3 保護方針
| バケット | 保護方式 |
| --- | --- |
| `hoge-orderhub-landing-prod` | Versioning、有効期限 30日 |
| `hoge-orderhub-processed-prod` | Versioning、90日保持 |
| `hoge-orderhub-status-prod` | Versioning、30日保持 |
| `hoge-orderhub-archive-prod` | Versioning、7年保持 |

## 5. リストア手順方針
### 5.1 RDS 論理障害
1. 障害発生時刻を確定する。
2. 影響範囲を確認し、PITR の復旧時点を決定する。
3. 一時復旧 DB を作成する。
4. 差分確認後、本番切替または必要データのみ抽出反映する。

### 5.2 S3 誤削除
1. 削除オブジェクトのキーを特定する。
2. Versioning から前世代を復元する。
3. 連携済み状態と整合性を確認する。

### 5.3 HULFT サーバ障害
1. EC2 の障害切り分けを行う。
2. 修復困難時は起動テンプレートから新規起動する。
3. HULFT 設定ファイル、ホスト ID、監視設定を復元する。
4. S3 正本から処理再開可否を確認する。

## 6. 定期訓練
| 項目 | 頻度 |
| --- | --- |
| RDS PITR 訓練 | 四半期ごと |
| S3 復元訓練 | 半期ごと |
| HULFT 再構築訓練 | 年1回 |
