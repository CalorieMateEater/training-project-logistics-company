# データストア設計書

## 1. 目的
Hoge社システムで利用する共通業務データストアおよび S3 バケット群の構成、接続、バックアップ、保持期間、暗号化方針を定義する。

## 2. データストア一覧
| 種別 | サービス | 用途 |
| --- | --- | --- |
| RDB | Amazon RDS for PostgreSQL | `orderhub`、`customer_registry`、`stockkeeper` の各スキーマで注文、配送状態、顧客、在庫、連携履歴を管理 |
| オブジェクト | Amazon S3 | 着信、処理済み、返却、アーカイブ |

## 3. RDS 設計
| 項目 | 内容 |
| --- | --- |
| エンジン | PostgreSQL 15 |
| 構成 | Multi-AZ |
| 接続方式 | RDS Proxy 経由 |
| インスタンスクラス | `db.m6g.large` |
| ストレージ | gp3 300GB |
| 保管時暗号化 | KMS |
| 転送時暗号化 | TLS 必須 |

### 3.1 スキーマ構成
| スキーマ | 主用途 | 主な利用サービス |
| --- | --- | --- |
| `orderhub` | 注文、出荷依頼、配送状態、通知履歴、連携履歴 | Hoge OrderHub Batch / Worker / Shipping Gateway |
| `customer_registry` | 顧客マスタ、会員状態 | Hoge Customer Registry |
| `stockkeeper` | 倉庫場所、在庫残高、引当台帳、入庫履歴、在庫トランザクション | Hoge Stock Keeper |

## 4. RDS 接続方針
| 項目 | 内容 |
| --- | --- |
| 最大アプリ接続 | 200 |
| 接続プール | RDS Proxy を利用 |
| 接続タイムアウト | 5秒 |
| クエリタイムアウト | 30秒 |

- RDS は共通インスタンスを採用するが、アプリケーションごとに接続ユーザを分離し、参照可能スキーマを限定する。
- `Customer Registry` と `Stock Keeper` も `RDS Proxy` 経由で接続し、接続先は共通 RDS の各専用スキーマとする。

## 6. S3 バケット設計
| バケット / プレフィックス | 用途 | Versioning | 初期ストレージクラス | ライフサイクル |
| --- | --- | --- | --- | --- |
| `hoge-orderhub-landing-prod` | Foo 着信ファイル | 有効 | Standard | 受信後 30日保持し削除。非現行版は 7日保持後に削除 |
| `hoge-orderhub-archive-prod/foo/processed/` | Foo 処理済みファイル | 有効 | Standard | 30日後に Standard-IA へ移行、90日保持後に削除。非現行版は 30日保持後に削除 |
| `hoge-orderhub-archive-prod/foo/error/` | Foo エラーファイル | 有効 | Standard | 30日後に Standard-IA へ移行、90日保持後に削除。非現行版は 30日保持後に削除 |
| `hoge-orderhub-status-prod` | Foo 返却ファイル | 有効 | Standard | 14日後に Standard-IA へ移行、30日保持後に削除。非現行版は 14日保持後に削除 |
| `hoge-orderhub-archive-prod` | 売上配送実績アーカイブ | 有効 | Standard | 30日後に Standard-IA、180日後に Glacier Flexible Retrieval、365日後に Glacier Deep Archive へ移行し、7年保持後に削除 |

### 6.1 ストレージクラス選定方針
- `landing` は受信直後の再処理と障害調査で即時参照するため `Standard` 固定とする。
- `foo/processed/`、`foo/error/` と `status` は短中期の参照可能性を見込み、一定期間後に `Standard-IA` へ移行して保管コストを抑制する。
- `archive` は長期保存が主目的のため、参照頻度低下に合わせて段階的に `Glacier` 系ストレージクラスへ移行する。
- `hoge-orderhub-archive-prod` では、`foo/processed/` と `foo/error/` にプレフィックス固有の90日保持ルールを適用し、売上配送実績アーカイブの7年保持ルールと分離する。
- すべてのバケットで Versioning を有効化し、現行版と非現行版で保持日数を分けて誤更新・誤削除復旧に対応する。

## 7. データ保持
| データ | 保持期間 |
| --- | --- |
| 注文・配送状態 | 3年 |
| 顧客マスタ | 有効期間中 + 退会後1年 |
| 在庫残高 | 現行値として常時保持 |
| 在庫トランザクション / 入庫履歴 | 3年 |
| 連携履歴 | 1年 |
| 監査ログ | 7年 |
| アーカイブファイル | 7年 |

## 8. 性能方針
- OrderHub テーブルは `order_id`、`shipment_request_id`、`partner_order_id`、`status_updated_at` に索引を付与する。
- 配送状態更新は `order_id + status_seq` で一意制御する。
- Customer Registry は `customer_id` を主キーとし、参照系のみのため追加索引は最小限とする。
- Stock Keeper は `item_code + warehouse_location_code`、`warehouse_location_code + receipt_reference_no` を主な検索・排他キーとする。
- アーカイブ出力は夜間帯に限定し、業務本線と IO 競合しないようにする。
