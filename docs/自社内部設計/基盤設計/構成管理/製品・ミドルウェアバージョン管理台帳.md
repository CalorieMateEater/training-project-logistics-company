# 製品・ミドルウェアバージョン管理台帳

## 1. 目的
本書は、Hoge社システムで採用するアプリケーション、ミドルウェア、AWSマネージドサービス、運用エージェントのバージョン管理方針と採用バージョンを定義する。

構成変更、脆弱性対応、保守期限管理、障害調査時の切り分けにおいて、本台帳を基準資料として利用する。

## 2. 適用範囲
- Hoge OrderHub
- 配送連携APIシステム
- 顧客マスタ管理
- 在庫管理
- Foo社向けHULFT連携基盤
- AWS実行基盤、データ基盤、ネットワーク基盤

## 3. 管理方針
### 3.1 管理区分
| 区分 | 内容 | 管理方法 |
| --- | --- | --- |
| アプリケーション依存製品 | Spring Boot、JDK、Gradle、ライブラリなど | 明示的に採用バージョンを固定し、アップグレード時は設計書とビルド定義を同時更新する |
| 実行基盤依存製品 | ECS Fargate Platform Version、HULFT、CloudWatch Agent など | 環境ごとの採用バージョンを固定し、変更時は疎通・監視・性能影響を確認する |
| AWSマネージドサービス | ALB、VPC、S3、SQS など | 個別の製品バージョンは持たないため、AWS仕様変更の影響確認日と利用機能を管理する |
| データベースエンジン | PostgreSQL エンジンなど | メジャーバージョン/マイナーバージョン方針、保守期限、アップグレード方式を管理する |

### 3.2 更新契機
- セキュリティ脆弱性対応
- 保守期限到来
- AWS/EOL通知による更新
- 外部接続先要件変更
- Java/Spring Boot等の開発標準更新

### 3.3 変更管理ルール
- 本番反映前に検証環境でアプリ疎通、HULFT送受信、Bar/Fuga向けAPI疎通、SQS処理、日次アーカイブ処理を確認する。
- メジャーバージョン更新は、関連する基盤設計書、運用設計書、リリース設計書を同時に更新する。
- AWSマネージドサービスで顧客固定バージョンがないものは、利用機能、設計前提、確認日を記録する。

## 5. バージョンを持たないAWSマネージドサービスの管理方法
ALB、VPC、S3、SQS、CloudWatch Logs などのAWSマネージドサービスは、一般的なオンプレ製品のような利用者固定バージョンを持たない。

このため、本システムでは以下をバージョン相当の管理対象とする。

| 管理観点 | 具体例 |
| --- | --- |
| 利用機能 | ALBのTLSポリシー、S3バージョニング有無、SQS FIFO有無 |
| 設計前提 | Private ALB利用、VPC Endpoint利用、S3 SSE-KMS利用 |
| 影響確認日 | AWS仕様変更通知を確認した日付、再評価日 |
| 変更履歴 | 設定変更チケット番号、リリース番号、設計書更新履歴 |

## 6. 更新手順
1. 更新対象の脆弱性情報、保守期限、AWS通知を確認する。
2. 影響する設計書、ジョブ、API、外部連携を洗い出す。
3. 検証環境で単体疎通、結合疎通、バッチ処理、HULFT送受信、配送状態照会を確認する。
4. 本台帳と関連設計書を更新し、変更管理票へ記録する。
5. 本番反映後にバージョン確認結果を運用引継ぎ資料へ反映する。
