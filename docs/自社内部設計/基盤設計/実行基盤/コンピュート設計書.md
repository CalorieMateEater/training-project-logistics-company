# コンピュート設計書

## 1. 目的
Hoge社受注連携基盤における ECS、EC2、ECR、デプロイ、スケーリング、ヘルスチェック方針を定義する。

## 2. 実行基盤一覧
| コンポーネント | 基盤 | 配置 |
| --- | --- | --- |
| FooOrderImportJob | ECS Fargate Task（HULFT受信完了イベント起動） | 2AZ 対応クラスタ |
| 配送会社連携Worker | ECS Fargate Service | 2AZ |
| 配送結果受付API / 配送状態取込Worker | ECS Fargate Service | 2AZ |
| 配送結果返却Worker | ECS Fargate Service | 2AZ |
| 日次アーカイブBatch | ECS Fargate Scheduled Task | 2AZ 対応クラスタ |
| 出荷依頼受付 API / 出荷状態照会 API | ECS Fargate Service | 2AZ |
| Customer Registry | ECS Fargate Service | 2AZ |
| Stock Keeper | ECS Fargate Service | 2AZ |
| HULFT受信/送信 | EC2 | Protected サブネット |

## 3. ECS サービス設計
| サービス | Desired | Min | Max | 備考 |
| --- | ---: | ---: | ---: | --- |
| 配送会社連携Worker | 2 | 2 | 4 | CPU 70% でスケールアウト |
| 配送結果受付API / 配送状態取込Worker | 2 | 2 | 4 | リクエスト数基準。API受信と状態反映は同一サービス内で論理分離 |
| 配送結果返却Worker | 2 | 1 | 3 | キュー蓄積量で増減 |
| 出荷依頼受付 API | 2 | 2 | 6 | ALB RequestCountPerTarget 基準 |
| 出荷状態照会 API | 2 | 2 | 6 | ALB RequestCountPerTarget 基準 |
| Customer Registry | 2 | 2 | 4 | 社内呼出専用 |
| Stock Keeper | 2 | 2 | 6 | 社内呼出および倉庫担当者端末からの照会・入庫登録に対応 |

### 3.1 非機能性能要件への対応
| 対象 | 要求性能 | スケーリング指標 | 設計上限・確認方法 |
| --- | --- | --- | --- |
| 出荷依頼受付 API | 30 TPS、P95 1.5秒以内 | ALB `RequestCountPerTarget`、CPU 70% | 最大6タスクまで拡張し、200同時接続を含む負荷試験で確認する |
| 出荷状態照会 API | 50 TPS、P95 1.0秒以内 | ALB `RequestCountPerTarget`、CPU 70% | 最大6タスクまで拡張し、参照系ピーク負荷試験で確認する |
| 配送会社連携Worker | Bar連携 20 TPS | SQS滞留件数、最古メッセージ経過時間、CPU 70% | 最大4タスクまで拡張し、Bar側同時接続上限20を超えないよう送信並列数を制御する |
| 配送結果受付API / 配送状態取込Worker | 配送結果受信 20 TPS | ALBリクエスト数、内部処理滞留件数、CPU 70% | 最大4タスクまで拡張し、順序制御を維持した負荷試験で確認する |
| FooOrderImportJob | 1ファイル30秒以内、日量5万件 | EventBridge/ECS起動失敗数、実行中タスク数 | 別ファイルは並列実行し、アカウント上限とDB負荷を考慮した最大同時実行数を負荷試験で確定する |

- 性能試験では通常負荷、ピーク負荷、200同時接続、単一AZ縮退時を確認する。
- 性能要件を満たさない場合は、タスク上限、CPU/メモリ、RDS Proxy接続プール、DB索引を見直してから本番移行する。

## 4. デプロイ方式
| 対象 | デプロイ方式 |
| --- | --- |
| 外部 API | Blue/Green |
| 社内 API | Rolling |
| Worker | Rolling |
| FooOrderImportJob | イメージ差替え後、次回受信イベント起動から有効 |
| 日次アーカイブBatch | イメージ差替え後、次回スケジュールから有効 |
| HULFT EC2 | AMI 更新 + 起動テンプレート更新 |

## 5. ヘルスチェック
| 対象 | 方式 | 判定 |
| --- | --- | --- |
| 外部 API | ALB `/healthz` | 200 応答 2/2 |
| 社内 API | ALB `/healthz` | 200 応答 2/2 |
| Worker | ECS タスクヘルス | プロセス起動確認 |
| Batch | 実行結果コード | 0 を正常 |
| HULFT | プロセス監視 | HULFT デーモン起動確認 |

## 6. コンテナ設計
- ベースイメージは Amazon Linux 系の社内標準イメージを利用する。
- イメージ保管は ECR を利用する。
- Push 時に ECR 脆弱性スキャンを実施する。
- 標準出力ログは CloudWatch Logs に送信する。

## 7. タイムアウト / リトライ方針
- HTTP 通信のタイムアウトは [通信/HTTP通信設計書](../通信/HTTP通信設計書.md) に従う。
- DB 接続タイムアウトは 5 秒、クエリタイムアウトは 30 秒とする。
- Batch の外部 API 再送は冪等条件を満たす場合に限定する。

## 8. 補足
- 単一リージョン構成のため、リージョン障害対策は本設計対象外とする。
- HULFT は ECS 化せず、製品要件に合わせて EC2 運用とする。
- 配送結果受付APIと配送状態取込Workerは同一コンテナイメージを利用するが、受信エンドポイント処理と業務状態反映処理はアプリケーション責務として分離する。
