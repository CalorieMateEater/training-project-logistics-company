# オンライン処理方式設計書

## 1. 目的
本書は、Hoge社のオンライン処理に該当するAPI系処理について、処理方式、負荷分散、データ保護方式、タイムアウト時の挙動、コンテナ設計、コンテナリソースを定義する。

## 2. 対象処理
| 処理ID | 処理名 | 役割 | 入口 |
| --- | --- | --- | --- |
| `OL-001` | 出荷依頼受付API | Hoge社業務部門からの直受注受付 | `POST /api/v1/shipment-requests` |
| `OL-002` | 出荷状態照会API | Foo社 / Hoge社業務担当からの状態照会受付 | `GET /api/v1/shipment-status/{lookup_key}` |
| `OL-003` | 配送結果受付API | Bar社 / Fuga社からの配送イベント受付 | `POST /api/v1/delivery-results/bar`, `POST /api/v1/delivery-results/fuga` |
| `OL-004` | 顧客確認API | OrderHubからの顧客参照 | `GET /api/v1/customers/{customer_id}/status` |
| `OL-005` | 在庫引当API | OrderHubからの在庫引当要求 | `POST /api/v1/stocks/reservations` |
| `OL-007` | 在庫引当解除API | OrderHubからの引当解除要求 | `POST /api/v1/stocks/reservations/{reservation_id}/releases` |
| `OL-008` | 在庫出荷確定API | OrderHubからの出荷確定要求 | `POST /api/v1/stocks/reservations/{reservation_id}/ship-confirms` |
| `OL-009` | 倉庫在庫照会API | 倉庫担当者からの在庫照会受付 | `GET /api/v1/stocks/inventories` |
| `OL-010` | 入庫登録API | 倉庫担当者からの入庫登録受付 | `POST /api/v1/stocks/receipts` |
| `OL-006` | 配送状態反映Worker | 配送結果受付APIから引き継いだBar/Fugaイベントの状態反映 | 配送結果受付API受信完了 |

## 3. 処理方式
| 処理ID | 同期/非同期 | トランザクション境界 | 主な後続処理 |
| --- | --- | --- | --- |
| `OL-001` | 同期 | 注文起票、顧客確認結果、在庫引当結果、出荷依頼起票を1トランザクションで確定 | `bar-shipment-request-queue.fifo` または `fuga-shipment-request-queue.fifo` へ送信要求投入 |
| `OL-002` | 同期 | 読み取り専用。更新は連携履歴のみ | 最新注文状態と配送状態を返却 |
| `OL-003` | 同期 | 受付監査、相関ID採番、状態反映要求の引継を1トランザクションで確定 | 配送状態反映Worker の起動 |
| `OL-004` | 同期 | 読み取り専用 | 顧客状態応答 |
| `OL-005` | 同期 | 倉庫場所確定、引当済在庫更新、引当結果登録を同一要求内で完結 | 引当結果応答 |
| `OL-007` | 同期 | 引当済在庫解除と監査記録を同一要求内で完結 | 引当解除結果応答 |
| `OL-008` | 同期 | 出荷確定、保有在庫減算、監査記録を同一要求内で完結 | 出荷確定結果応答 |
| `OL-009` | 同期 | 読み取り専用。認可判定後に在庫残高を参照する | 在庫一覧応答 |
| `OL-010` | 同期 | 入庫受付履歴重複判定、在庫更新、監査記録を同一要求内で完結 | 更新後在庫応答 |
| `OL-006` | 非同期 | 配送状態履歴登録、最新状態更新、通知要求起票を1トランザクションで確定 | Foo返却用通知要求の作成 |

## 4. 負荷分散・可用化方式
| 項目 | 方式 |
| --- | --- |
| AZ分散 | すべてのECSサービスを2AZへ分散配置する |
| 入口負荷分散 | 対外接続APIは Private ALB、社内APIは Internal ALB で負荷分散する |
| ターゲット分散 | ALB Target Group に複数タスクを登録し、ラウンドロビンで分散する |
| ヘルスチェック | `/healthz` へ 200 応答を返すことをもって正常とする |
| 障害時切替 | タスク異常時はECSが再配置し、片系ALBターゲット異常時は健全タスクへ自動切替する |

## 5. データ保護方式
| 観点 | 方式 | 補足 |
| --- | --- | --- |
| 通信保護 | VPN閉域網 + HTTPS/TLS | Foo/Bar/Fugaとの外部通信は閉域接続前提 |
| 秘密情報保護 | Secrets Manager + KMS | APIキー、DB認証情報、OAuth資格情報を保護 |
| データ整合性 | RDSトランザクション制御 | API単位でコミット/ロールバックを明確化する |
| 再実行整合 | `partner_request_id`、`reservation_id`、`status_seq` による再試行判定 | 注文登録確定前失敗時に重複確保、重複解除、重複反映を避ける |
| 冪等保護 | `X-Idempotency-Key`、`partner_request_id`、`status_seq`、`receipt_reference_no` | 重複送信、重複受付、逆順イベント、重複入庫登録を制御する |
| 在庫排他 | `item_code + warehouse_location_code` 単位の排他制御 | 同一倉庫在庫の同時更新競合を防止する |
| 利用者認可 | `employee_id + warehouse_location_code` の担当スコープ判定 | 倉庫担当者の参照・登録範囲を制御する |
| 監査保護 | 連携履歴、通知履歴、CloudWatch Logs | IF単位追跡を可能にする |
| 個人情報保護 | アプリログのマスキング | 住所、電話番号は全文をログ出力しない |

## 6. 接続タイムアウト時の挙動
| 処理ID | 接続/読込タイムアウト | タイムアウト時の挙動 | 再送方針 |
| --- | --- | --- | --- |
| `OL-001` | 2秒 / 10秒 | DB未確定の場合はロールバックし、`503` または `504` 相当で応答する。同一 `partner_request_id` の再試行で整合回復する | 呼出元は最大1回再送 |
| `OL-002` | 2秒 / 5秒 | 応答不能時は `503` 相当で応答し、参照処理として副作用は残さない | 呼出元は最大1回再送 |
| `OL-003` | 2秒 / 15秒 | Bar/Fuga側は再送対象とし、Hoge側は `status_seq` により重複吸収する | Bar/Fuga側最大2回再送 |
| `OL-004` | 1秒 / 3秒 | 呼出元へタイムアウト例外を返し、注文取込/API受付側で失敗扱いにする | 呼出元最大2回再送 |
| `OL-005` | 1秒 / 5秒 | 呼出元へタイムアウト例外を返し、在庫未確定として失敗扱いにする | 呼出元最大1回再送 |
| `OL-007` | 1秒 / 5秒 | 呼出元へタイムアウト例外を返し、引当解除未確定として扱う | 呼出元最大1回再送 |
| `OL-008` | 1秒 / 5秒 | 呼出元へタイムアウト例外を返し、出荷確定未確定として扱う | 呼出元最大1回再送 |
| `OL-009` | 2秒 / 5秒 | 応答不能時は `503` 相当で応答し、副作用は残さない | 呼出元最大1回再送 |
| `OL-010` | 2秒 / 10秒 | 在庫更新未確定として扱い、同一 `receipt_reference_no` での再送で整合回復する | 呼出元自動再送なし、利用者再試行 |
| `OL-006` | 1秒 / 15秒 | 最新状態更新前に失敗した場合は未反映として再処理対象に残し、`status_seq` により再実行時も重複反映しない | 内部再実行 3回 |

## 9. 補足
- 対外接続APIはインターネット公開ではなく、閉域VPN配下のPrivate ALBで受ける。
- 出荷依頼受付APIは同期APIだが、配送会社連携自体は `bar-shipment-request-queue.fifo` または `fuga-shipment-request-queue.fifo` 経由の非同期処理に委譲する。
- 配送結果受付APIはBar社・Fuga社からのイベント受信点であり、業務状態反映は配送状態取込Workerが担う。
- 配送結果受付APIと配送状態取込Workerは同一ECSサービスに同居させるが、API受信層と状態反映層はアプリケーション責務として分離する。
- Stock Keeper は在庫管理サービスとして、在庫引当、引当解除、出荷確定、在庫照会、入庫登録を同一サービスで提供する。
