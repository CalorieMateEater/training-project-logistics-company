# 非機能要件定義書

## 1. 目的
Hoge社の受注連携基盤で採用する可用性、復旧目標、性能、セキュリティ、運用、コスト、保守性の基準を定義する。

## 2. 非機能要件一覧
| 観点 | 要件 |
| --- | --- |
| 可用性 | 受注登録、出荷依頼、状態照会の基幹機能は月間稼働率 `99.9%` 以上とする |
| 単一AZ障害耐性 | ECS API、Worker、RDS、S3、SQS は単一 AZ 障害時も継続可能とする。HULFT は代替インスタンス起動により復旧する |
| RTO | API 系 30 分以内、Batch/HULFT 系 60 分以内 |
| RPO | OrderHub データストア 5 分以内、S3 保管ファイル 0 分、HULFT ローカル一時領域 24 時間以内再取得可能 |
| 性能 | 出荷依頼 API 30 TPS、状態照会 API 50 TPS、Bar 連携 20 TPS、配送結果受信 20 TPS |
| 同時接続 | 社内外 API 合計 200 同時接続まで劣化なく処理する |
| レスポンスタイム | 状態照会 API 95 パーセンタイル 1.0 秒以内、出荷依頼 API 95 パーセンタイル 1.5 秒以内 |
| バッチ処理時間 | Foo 注文取込は 1 ファイル 30 秒以内かつ日量 5 万件処理可能、日次アーカイブ 60 分以内 |
| セキュリティ | 外部公開は行わず、閉域接続 + mTLS + API キーを基本とする |
| 運用 | CloudWatch メトリクス、ログ、アラート、Runbook を整備する |
| コスト | 月額上限 100 万円、70/90/100% で予算アラート通知する |
| 保守性 | Terraform による IaC 管理、環境差分は変数とパラメータで吸収する |

## 3. RTO/RPO 詳細
| 対象 | RTO | RPO | 備考 |
| --- | --- | --- | --- |
| 出荷依頼受付 API / 出荷状態照会 API | 30分 | 5分 | ECS 再配置、RDS Multi-AZ フェイルオーバーを前提 |
| 配送会社連携Worker / 配送状態取込Worker | 30分 | 5分 | タスク再起動で復旧 |
| Foo 注文取込 HULFT | 60分 | 0分 | S3 着信済みであれば再取込可能 |
| 注文ステータス返却 HULFT | 60分 | 15分 | S3 上の返却ファイル再送で復旧 |
| 日次アーカイブ | 翌営業日開始まで | 24時間 | 業務本線外 |

## 4. 採用判断
- マルチリージョンは採用しない。
- 単一リージョン内での Multi-AZ を基本とする。
- 外部接続はすべて閉域網とし、インターネット公開前提の SLA は要求しない。

## 5. 性能前提
| 項目 | 想定値 |
| --- | --- |
| 1日あたり注文件数 | 50,000件 |
| ピーク時間帯注文比率 | 1時間あたり 20% |
| 配送結果返却件数 | 1日 80,000イベント |
| 1メッセージサイズ | 4KB 以内 |
| S3 1日入出力ファイル数 | 受信 50,000 本以内、受付通知 50,000 本以内、配送結果返却 80,000 本以内、アーカイブ 1 本 |

- Foo社注文は注文確定都度に HULFT ファイル送信される前提とする。
- Foo社向け注文受付通知、および配送結果返却ファイルもイベント単位で都度生成する。

## 6. 運用・保守前提
- 監視一次受けは運用当番が担当する。
- 本番変更は Pull Request 承認後に CI/CD パイプラインで反映する。
- アプリケーション設定値は SSM Parameter Store または Secrets Manager で管理する。
