# 業務ルール定義書

## 1. 目的
本書は、Hoge社の出荷・配送管理における主要な業務ルールを定義し、受注受付、配送条件判定、Bar社連携、配送状態管理、対外返却、状態照会、アーカイブまでの判断基準を整理することを目的とする。

## 2. 適用範囲
- Foo社からの HULFT 出荷依頼ファイル受付
- Fuga社からの HTTPS 出荷依頼受付
- Foo社 / Fuga社からの HTTPS 状態照会
- Hoge社内の顧客確認、在庫引当、配送条件判定
- Bar社向け出荷依頼送信
- Bar社からの配送結果受信
- Foo社向け配送結果返却
- Baz社 / Qux社向け非同期通知
- 日次アーカイブ

## 4. 用語
| 用語 | 定義 |
| --- | --- |
| 注文元 | Hoge社が受け付けた依頼の流入元。`FOO` または `FUGA` |
| 提携先優先度 | Foo社が連携する `priority_level` の数値。0〜9 |
| 優先配送区分 | Bar社向け送信時の優先区分。`NORMAL` または `PRIORITY` |
| 出荷解放日時 | 予約注文・予約出荷を配送会社へ引き渡してよい日時 |
| 送信待ちキュー | Bar社営業時間に合わせて送信待機に利用する内部キュー |
| 配送状態 | Bar社から返却される配送イベントにもとづく現在状態 |
| 理由分類コード | Bar社から返却される配送例外・補正状態の外部表示用分類 |
| 配送イベント連番 | Bar社から通知される配送結果の順序を表す連番 |
| 注文受付通知 | Hoge社が Foo社へ返却する「Hoge社内で注文受付済」を示すファイル |

## 5. ルール体系
| 分類 | 内容 |
| --- | --- |
| 受注受付ルール | 受注元登録、受付条件、受付通知 |
| 優先配送ルール | Foo優先度の保持、Bar向け変換 |
| 配送会社連携ルール | 配送条件判定、営業時間制約、送信制御 |
| 配送状態管理ルール | 状態更新、順序制御、例外状態 |
| 対外通知ルール | Foo返却、Baz/Qux通知、状態照会 |
| 保守運用ルール | アーカイブ、保持、責任分界 |

## 6. 受注受付ルール

### 6.1 受注元登録
| ルールID | ルール |
| --- | --- |
| `BR-001` | Foo社の HULFT ファイルから受け付けた注文は、注文元を Foo として登録する。 |
| `BR-002` | Fuga社の出荷依頼受付 API から受け付けた注文は、注文元を Fuga として登録する。 |
| `BR-003` | 注文元は受付時点で確定し、後続処理で別値へ更新しない。 |
| `BR-004` | Foo社からの HULFT ファイルは注文確定都度に送信され、Hoge社は受信完了をトリガに即時取り込む。 |
| `BR-005` | Foo社予約注文またはFuga社予約出荷で `shipping_release_at` が未来日時の場合、顧客確認・在庫引当後に出荷解放待ち状態で保持する。 |

### 6.2 受付単位と重複
| ルールID | 条件 | 処理 | 影響データ |
| --- | --- | --- | --- |
| `BR-011` | Foo社ファイル受信 | 1明細レコードを1受注単位として取り込む | 注文ヘッダ、注文明細 |
| `BR-012` | Fuga API受付 | 1 API リクエストを1受注単位として取り込む | 注文ヘッダ、注文明細、出荷依頼 |
| `BR-013` | Fuga APIで `partner_request_id` 重複 | 新規受付せず重複エラーとする | 連携履歴 |

### 6.3 注文受付通知
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-021` | Foo注文の基本バリデーション、顧客確認、在庫引当が正常終了 | Foo社へ注文受付通知ファイルを返却する |
| `BR-022` | 注文受付通知は、Bar社への送信成功を意味しない | `RECEIVED` 相当の受付完了のみ通知する |
| `BR-023` | Bar社営業時間外で送信待機中でも、Foo社への注文受付通知は返却対象とする | 送信待ちと受付完了は別概念として扱う |
| `BR-024` | `shipping_release_at` が未来日時の Foo注文では、受付通知を `RECEIVED_HOLD` として返却する | Bar社未送信でも受付自体は成功として扱う |

## 7. 入力バリデーションルール

### 7.1 Foo社ファイル
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-101` | `record_type != D` | エラー終了 |
| `BR-102` | `order_type` が `01`,`02`,`03` 以外 | エラー終了 |
| `BR-103` | `priority_level` が 0〜9 の整数以外 | エラー終了 |
| `BR-104` | `order_type != 02` かつ `priority_level != 0` | エラー終了 |
| `BR-105` | `customer_id` が顧客マスタに存在しない | エラー終了 |
| `BR-106` | `item_code` が在庫管理上無効 | エラー終了 |

### 7.2 Fuga社 API
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-111` | 認証情報不正、接続元不正 | 受付拒否 |
| `BR-112` | 顧客 ID 不存在 | 422 エラー |
| `BR-113` | 商品コード無効、数量不正 | 422 エラー |
| `BR-114` | 配送希望日が当日より前 | 422 エラー |
| `BR-115` | `shipment_mode=RESERVED` かつ `shipping_release_at` 未指定 | 422 エラー |

## 8. 優先配送ルール

### 8.1 優先度保持
| ルールID | ルール |
| --- | --- |
| `BR-201` | Foo注文では `priority_level` を `partner_priority_level` として保持する。 |
| `BR-202` | Fuga注文では `partner_priority_level` を保持せず、実質 `0` 扱いとする。 |
| `BR-203` | Hoge社は提携先優先度を保持するが、その数値の意味づけを自社ルールで再計算しない。 |

### 8.2 Bar向け優先配送区分変換
| ルールID | 条件 | Hoge登録値 | Bar向け値 |
| --- | --- | --- | --- |
| `BR-211` | Foo注文かつ `order_type=02` かつ `priority_level>=5` | `order_source=FOO`、`partner_priority_level=5..9` | `shipping_priority_class=PRIORITY` |
| `BR-212` | Foo注文かつ上記以外 | `order_source=FOO`、`partner_priority_level=0..4` | `shipping_priority_class=NORMAL` |
| `BR-213` | Fuga注文 | `order_source=FUGA` | `shipping_priority_class=NORMAL` |

### 8.3 責任分界
| ルールID | ルール |
| --- | --- |
| `BR-221` | Hoge社は `PRIORITY` / `NORMAL` の区分を Bar社向けに送信するが、Bar社内の配送資材引当や処理順ロジックは管理しない。 |
| `BR-222` | Foo優先注文が Fuga注文より先に処理される可能性はあるが、その可否判定は Bar社責務とする。 |

## 9. 配送会社連携ルール

### 9.1 配送条件判定
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-301` | 顧客確認、在庫引当が正常終了 | 配送条件判定へ進む |
| `BR-302` | 本システムの現行スコープで配送会社連携先が必要 | Bar社向け出荷依頼を生成する |
| `BR-303` | 出荷依頼生成時 | 出荷依頼識別子を採番し、出荷依頼状態を送信待ちとする |
| `BR-304` | `shipping_release_at` が未来日時 | Bar社営業時間内であっても解放日時到来前は送信しない |

### 9.2 Bar社営業時間制約
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-311` | Bar社営業時間 平日08:00-18:00 内 | Hoge社は送信待ちキューから取得し Bar社へ出荷依頼を送信する |
| `BR-312` | Bar社営業時間外 | 送信失敗扱いにせず営業時間待ち状態のまま待機する |
| `BR-313` | タイムアウトまたは 5xx | 再送対象として再送待ち状態とし、営業時間内に再送する |
| `BR-314` | 4xx 業務エラー | 自動再送せず業務エラーとして扱う |

### 9.3 送信方式
| ルールID | ルール |
| --- | --- |
| `BR-321` | Hoge社から Bar社への出荷依頼は、同期 HTTPS API で送信する。 |
| `BR-322` | ただし呼出契機は受付処理と分離し、送信待ちキュー経由の非同期処理とする。 |
| `BR-323` | Hoge社は同一配送依頼の重複送信を防止する。 |

## 10. 配送状態管理ルール

### 10.1 配送結果受信
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-401` | Bar社から配送結果通知受信 | 配送状態履歴へ記録し、配送状態最新を更新判定する |
| `BR-402` | 同一配送依頼で配送イベント連番重複 | 再送として扱い、重複登録しない |
| `BR-403` | 受信した配送イベント連番が保持済みより小さい | 旧イベントとして無視する |
| `BR-404` | 受信した配送イベント連番が保持済みより大きい | 最新状態を更新する |
| `BR-405` | `reason_category` または住所補正情報が付与されている | 配送状態最新と履歴へ保持し、対外表示用状態名へ反映する |

### 10.2 注文状態・配送状態反映
| ルールID | 条件 | 更新内容 |
| --- | --- | --- |
| `BR-411` | Bar受付済受信 | 注文状態を配送受付済へ更新 |
| `BR-412` | 配送準備中受信 | 注文状態を配送準備中へ更新 |
| `BR-413` | 配送中受信 | 注文状態を配送中へ更新 |
| `BR-414` | 配送完了受信 | 注文状態を配送完了へ更新 |
| `BR-415` | 配送失敗、持戻り、住所不備受信 | 注文状態を配送例外へ更新 |
| `BR-416` | 再配達待ち受信 | 注文状態を再配達待ちへ更新 |
| `BR-417` | キャンセル受信 | 注文状態をキャンセルへ更新 |

### 10.3 終端制御
| ルールID | ルール |
| --- | --- |
| `BR-421` | 配送完了、キャンセルは終端状態とし、後続で送信待ちや配送中へ戻さない。 |
| `BR-422` | 終端後の状態更新は、配送イベント連番と業務妥当性の両方を満たす場合のみ例外的に許容し、通常は受け付けない。 |

## 11. 対外通知ルール

### 11.1 Foo社向け配送結果返却
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-501` | 配送状態が返却対象へ変化 | Foo社向け配送結果返却ファイルを生成する |
| `BR-502` | 返却先は Foo社のみ | 配送結果返却ファイルは Foo社向けに限定する |
| `BR-503` | 返却は HULFT で実施 | HTTPS API では返却しない |

### 11.2 Baz社 / Qux社向け非同期通知
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-511` | Bar向け出荷依頼送信成功 | Baz社向けに請求予定通知を行う |
| `BR-512` | Bar向け出荷依頼送信成功 | Qux社向けに注文状態通知を行う |
| `BR-513` | 配送完了受信 | Baz社向けに確定請求通知を行う |
| `BR-514` | 配送状態更新受信 | Qux社向けに通知キー付き状態更新通知を行う |
| `BR-515` | Qux社向け通知 | Hoge社は夜間抑止せず即時送信し、受信側表示制御へ委譲する |

### 11.3 状態照会 API
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-521` | Foo社または Fuga社から状態照会 API 呼出 | 最新の注文状態、配送状態を返却する |
| `BR-522` | 同一 `partner_order_id` に複数配送イベントが存在 | 最新イベントを返却する |
| `BR-523` | Fuga社から `partner_request_id` で照会 | 同一注文へ解決できる場合は最新状態を返却する |
| `BR-524` | 状態照会 API は閉域 VPN 上の HTTPS で提供する | 所定の認証と接続元制御で保護する |
| `BR-525` | 状態照会結果には、受付済、出荷解放待ち、送信待ち、Bar受付済、配送中、配送完了などの中間状態を含める | 最終状態のみには限定しない |

## 12. アーカイブルール
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-601` | 日次アーカイブBatch 起動 | 完了済み注文、配送結果、請求連携履歴、通知履歴を抽出する |
| `BR-602` | 抽出対象データあり | アーカイブ出力 S3 へ出力し、売上アーカイブ S3 へ保管する |
| `BR-603` | アーカイブは業務本線と分離した保守処理として扱う | 業務同期応答に影響させない |

## 13. ルール間の優先順位
1. 認証・入力バリデーションルール
2. 注文元登録ルール
3. 顧客確認、在庫引当ルール
4. 優先配送区分変換ルール
5. 配送条件判定ルール
6. Bar営業時間制約ルール
7. 配送状態更新ルール
8. 対外返却、状態照会、アーカイブルール

## 14. 設計への引継ぎ方針
- 本書で定義した受付条件、優先配送条件、営業時間制約、状態更新条件をもとに、業務設計およびIF設計で実現方法を具体化する。
- 技術的な実装方式、ヘッダ項目、キュー名、内部テーブル更新順は設計資料で定義する。
