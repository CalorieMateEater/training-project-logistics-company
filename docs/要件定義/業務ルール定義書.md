# 業務ルール定義書

## 1. 目的
本書は、Hoge社の出荷・配送管理における主要な業務ルールを定義し、受注受付、配送条件判定、Bar社連携、配送状態管理、対外返却、状態照会、アーカイブまでの判断基準を整理することを目的とする。

## 2. 適用範囲
- Foo社からの HULFT 出荷依頼ファイル受付
- Hoge社業務部門からの内部出荷依頼受付
- Foo社からの HTTPS 状態照会
- Hoge社内の顧客確認、在庫引当、配送条件判定
- Hoge社倉庫担当者による在庫照会
- Hoge社倉庫担当者による入庫登録
- Bar社向け出荷依頼送信
- Fuga社向け出荷依頼送信
- Bar社からの配送結果受信
- Fuga社からの配送結果受信
- Foo社向け配送結果返却
- Baz社 / Qux社向け非同期通知
- 日次アーカイブ

## 4. 用語
| 用語 | 定義 |
| --- | --- |
| 荷主 | 商品を保有し、出荷可否と在庫管理責任を持つ主体。本システムでは Hoge社 |
| 配送委託先 | Hoge社から配送業務を委託され、実配送を行う外部配送会社。本システムでは Bar社、Fuga社 |
| 保有在庫数 | Hoge社が倉庫場所単位で物理的に保有する在庫数 |
| 引当済在庫数 | 受注に紐づき確保済みで、まだ出荷確定されていない在庫数 |
| 利用可能在庫数 | 新規引当に利用できる在庫数。`保有在庫数 - 引当済在庫数` |
| 倉庫場所コード | 在庫が保管される倉庫・保管拠点を識別するコード |
| 在庫引当解除 | 引当済在庫を保有在庫へ戻し、再利用可能にする処理 |
| 在庫出荷確定 | 引当済在庫を実出荷扱いとし、保有在庫数と引当済在庫数へ反映する処理 |
| 入庫 | 倉庫担当者が実際に倉庫へ到着した商品数量を在庫へ加算する処理 |
| 担当倉庫場所 | 倉庫担当者に割り当てられた参照・入庫登録可能な倉庫場所コード群 |
| 入庫受付番号 | 倉庫担当者が入庫登録時に指定する重複防止用受付番号 |
| 注文元 | Hoge社が受け付けた依頼の流入元。`FOO` または `HOGE` |
| 提携先優先度 | Foo社が連携する `priority_level` の数値。0〜9 |
| 優先配送区分 | Bar社向け送信時の優先区分。`NORMAL` または `PRIORITY` |
| 出荷解放日時 | 予約注文・予約出荷を配送会社へ引き渡してよい日時 |
| 配送会社送信待ちキュー | 配送会社別の送信制約に応じて送信待機に利用する内部キュー。`bar-shipment-request-queue.fifo`、`fuga-shipment-request-queue.fifo` を含む |
| 配送状態 | Bar社から返却される配送イベントにもとづく現在状態 |
| 理由分類コード | Bar社から返却される配送例外・補正状態の外部表示用分類 |
| 配送イベント連番 | Bar社から通知される配送結果の順序を表す連番 |
| 注文受付通知 | Hoge社が Foo社へ返却する「Hoge社内で注文受付済」を示すファイル |

| 分類 | 内容 |
| --- | --- |
| 受注受付ルール | 受注元登録、受付条件、受付通知 |

| 優先配送ルール | Foo優先度の保持、Bar向け変換 |

| 保守運用ルール | アーカイブ、保持、責任分界 |

## 6. 受注受付ルール

### 6.1 受注元登録
| ルールID | ルール |
| --- | --- |
| `BR-001` | Foo社の HULFT ファイルから受け付けた注文は、注文元を Foo として登録する。 |
| `BR-002` | Hoge社業務部門が内部登録した注文は、注文元を Hoge として登録する。 |
| `BR-003` | 注文元は受付時点で確定し、後続処理で別値へ更新しない。 |
| `BR-004` | Foo社からの HULFT ファイルは注文確定都度に送信され、Hoge社は受信完了をトリガに即時取り込む。 |
| `BR-005` | Foo社予約注文またはHoge直受注で `shipping_release_at` が未来日時の場合、顧客確認・在庫引当後に出荷解放待ち状態で保持する。 |
| `BR-006` | 在庫引当は Hoge社保有在庫に対して実施し、配送会社在庫の引当は行わない。 |
| `BR-007` | 在庫は `商品コード + 倉庫場所コード` 単位で管理し、在庫引当時に対象倉庫場所を確定する。 |
| `BR-008` | 利用可能在庫数は `保有在庫数 - 引当済在庫数` で算出し、利用可能在庫数が不足する倉庫場所は引当対象外とする。 |

### 6.2 受付単位と重複
| ルールID | 条件 | 処理 | 影響データ |
| --- | --- | --- | --- |
| `BR-011` | Foo社ファイル受信 | 1明細レコードを1受注単位として取り込む | 注文ヘッダ、注文明細 |
| `BR-012` | Hoge直受注受付 | 1 内部登録要求を1受注単位として取り込む | 注文ヘッダ、注文明細、出荷依頼 |
| `BR-013` | Hoge直受注で `request_id` 重複 | 新規受付せず重複エラーとする | 連携履歴 |

### 6.3 注文受付通知
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-021` | Foo注文の基本バリデーション、顧客確認、在庫引当が正常終了 | Foo社へ注文受付通知ファイルを返却する |
| `BR-022` | 注文受付通知は、Bar社への送信成功を意味しない | `RECEIVED` 相当の受付完了のみ通知する |
| `BR-023` | Bar社営業時間外で送信待機中でも、Foo社への注文受付通知は返却対象とする | 送信待ちと受付完了は別概念として扱う |
| `BR-024` | `shipping_release_at` が未来日時の Foo注文では、受付通知を `RECEIVED_HOLD` として返却する | Bar社未送信でも受付自体は成功として扱う |

### 6.4 在庫ライフサイクル
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-031` | 在庫引当成功 | 保有在庫数は減算せず、引当済在庫数を加算する |
| `BR-032` | 引当解除成功 | 引当済在庫数を減算し、保有在庫数は変更しない |
| `BR-033` | 在庫出荷確定成功 | 保有在庫数と引当済在庫数を同時に減算する |
| `BR-034` | 在庫引当時 | 倉庫場所コードを引当結果へ保持し、後続の配送条件判定と出荷依頼へ引き継ぐ |
| `BR-035` | 在庫引当解除 | 在庫出荷確定前の引当に対してのみ実施できる |
| `BR-036` | 在庫出荷確定 | 初版では Bar社からの `配送受付済` イベントを受信した時点で実行する |

### 6.5 排他制御
| ルールID | ルール |
| --- | --- |
| `BR-041` | 同一 `商品コード + 倉庫場所コード` に対する在庫引当、引当解除、出荷確定は排他的に実行する。 |
| `BR-042` | 排他待ちがタイムアウトした場合、在庫処理は再送可能異常として扱い、呼出元で再試行または運用判断を行う。 |
| `BR-043` | 排他制御は在庫システム側で実施し、OrderHub側は在庫処理結果をもとに業務継続可否を判断する。 |

### 6.6 倉庫担当者の参照権限
| ルールID | ルール |
| --- | --- |
| `BR-051` | 倉庫担当者は、自身に割り当てられた担当倉庫場所の在庫のみ照会できる。 |
| `BR-052` | 複数の倉庫担当者が同一倉庫場所を担当してよい。 |
| `BR-053` | 倉庫在庫照会では、保有在庫数、引当済在庫数、利用可能在庫数、最終入庫日時を返却する。 |

### 6.7 入庫登録
| ルールID | ルール |
| --- | --- |
| `BR-054` | 入庫登録は `商品コード + 倉庫場所コード` 単位で行う。 |
| `BR-055` | 入庫成功時は、保有在庫数と利用可能在庫数を入庫数量分だけ加算し、引当済在庫数は変更しない。 |
| `BR-056` | 在庫残高が未作成であっても、商品コードと倉庫場所コードが有効なら新規在庫残高を作成できる。 |
| `BR-057` | 入庫登録には `入庫受付番号` を必須とし、同一担当倉庫場所内で重複する受付番号は二重計上しない。 |
| `BR-058` | 同一の `入庫受付番号` で同一内容の再送があった場合は、重複成功として同一結果を返却する。 |
| `BR-059` | 同一の `入庫受付番号` で数量や商品コードが異なる要求があった場合は、業務エラーとして受け付けない。 |
| `BR-060` | 0以下の入庫数量、および担当外倉庫への入庫登録は受け付けない。 |

## 8. 優先配送ルール

### 8.1 優先度保持
| ルールID | ルール |
| --- | --- |
| `BR-201` | Foo注文では `priority_level` を `partner_priority_level` として保持する。 |
| `BR-202` | Hoge直受注では `partner_priority_level` を保持せず、実質 `0` 扱いとする。 |
| `BR-203` | Hoge社は提携先優先度を保持するが、その数値の意味づけを自社ルールで再計算しない。 |

### 8.2 Bar向け優先配送区分変換
| ルールID | 条件 | Hoge登録値 | Bar向け値 |
| --- | --- | --- | --- |
| `BR-211` | Foo注文かつ `order_type=02` かつ `priority_level>=5` | `order_source=FOO`、`partner_priority_level=5..9` | `shipping_priority_class=PRIORITY` |
| `BR-212` | Foo注文かつ上記以外 | `order_source=FOO`、`partner_priority_level=0..4` | `shipping_priority_class=NORMAL` |
| `BR-213` | Hoge直受注 | `order_source=HOGE` | `shipping_priority_class=NORMAL` |

### 8.3 責任分界
| ルールID | ルール |
| --- | --- |
| `BR-221` | Hoge社は `PRIORITY` / `NORMAL` の区分を Bar社向けに送信するが、Bar社内の配送資材引当や処理順ロジックは管理しない。 |
| `BR-222` | Foo優先注文は配送委託先へ渡す優先配送区分へ反映するが、配送会社内の処理順序までは Hoge社責務としない。 |
| `BR-223` | Hoge社は自社在庫の引当結果をもとに配送依頼可否を判断し、配送委託先である Bar社またはFuga社には配送実行に必要な情報のみを引き渡す。 |
| `BR-224` | Hoge社は配送委託先選定の将来拡張に備え、倉庫場所コードを出荷依頼単位へ保持する。 |

## 9. 配送会社連携ルール

### 9.1 配送条件判定
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-301` | 顧客確認、在庫引当が正常終了 | 配送条件判定へ進む |
| `BR-302` | 標準配送条件に合致 | Hoge社が配送委託先である Bar社向け出荷依頼を生成する |
| `BR-303` | 冷蔵便、大型商品、遠方配送などの特殊配送条件に合致 | Hoge社が配送委託先である Fuga社向け出荷依頼を生成する |
| `BR-304` | 出荷依頼生成時 | 出荷依頼識別子を採番し、出荷依頼状態を送信待ちまたは即時送信対象として登録する |
| `BR-305` | `shipping_release_at` が未来日時 | Bar社営業時間内であっても解放日時到来前は送信しない |
| `BR-306` | 配送条件判定時 | 在庫引当で確定した倉庫場所コードを参照し、出荷依頼へ設定する |

### 9.2 Bar社営業時間制約
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-311` | Bar社営業時間 平日08:00-18:00 内 | Hoge社は送信待ちキューから取得し Bar社へ出荷依頼を送信する |
| `BR-312` | Bar社営業時間外 | 送信失敗扱いにせず営業時間待ち状態のまま待機する |
| `BR-313` | タイムアウトまたは 5xx | 再送対象として再送待ち状態とし、営業時間内に再送する |
| `BR-314` | 4xx 業務エラー | 自動再送せず業務エラーとして扱う |

### 9.3 Fuga社送信制約
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-315` | Fuga社向け特殊配送案件 | 配送条件判定完了後、Fuga向け送信待ちキューへ登録し、即時送信対象とする |
| `BR-316` | タイムアウトまたは 5xx | 再送対象として再送待ち状態とする |
| `BR-317` | 4xx 業務エラー | 自動再送せず業務エラーとして扱う |

### 9.4 送信方式
| ルールID | ルール |
| --- | --- |
| `BR-321` | Hoge社から Bar社への出荷依頼は、同期 HTTPS API で送信する。 |
| `BR-322` | Hoge社から Fuga社への出荷依頼も、同期 HTTPS API で送信する。 |
| `BR-323` | Bar社向け呼出契機は受付処理と分離し、Bar向け送信待ちキュー経由の非同期処理とする。 |
| `BR-324` | Fuga社向け呼出契機も受付処理と分離し、Fuga向け送信待ちキュー経由の非同期処理とする。 |
| `BR-325` | Hoge社は同一配送依頼の重複送信を防止する。 |
| `BR-326` | Bar社向け必須項目は、注文受付時の配送先・金額スナップショットとStock Keeperの商品属性スナップショットから編集する。 |
| `BR-327` | Foo注文のBar社向け`partner_order_id`にはFoo社注文ID、Hoge直受注には`partner_request_id`を設定する。 |

## 10. 配送状態管理ルール

### 10.1 配送結果受信
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-401` | Bar社またはFuga社から配送結果通知受信 | 配送状態履歴へ記録し、配送状態最新を更新判定する |
| `BR-402` | 同一配送依頼で配送イベント連番重複 | 再送として扱い、重複登録しない |
| `BR-403` | 受信した配送イベント連番が保持済みより小さい | 旧イベントとして無視する |
| `BR-404` | 受信した配送イベント連番が保持済みより大きい | 最新状態を更新する |
| `BR-405` | `reason_category` または住所補正情報が付与されている | 配送状態最新と履歴へ保持し、対外表示用状態名へ反映する |
| `BR-406` | Fuga社から同一`status_seq`かつ同一内容を再受信 | 冪等成功として`202 Accepted`を返し、状態反映要求を重複起票しない |
| `BR-407` | Fuga社から同一`status_seq`かつ異なる内容を受信 | 競合として`409 Conflict`を返す |
| `BR-408` | Fuga社から保持済みより小さい`status_seq`を受信 | 旧イベントとして`409 Conflict`を返し、状態反映要求を起票しない |
| `BR-409` | Fuga社から保持済みより大きい`status_seq`を受信 | 状態遷移が妥当なら`202 Accepted`を返し、状態反映要求を起票する |

### 10.2 注文状態・配送状態反映
| ルールID | 条件 | 更新内容 |
| --- | --- | --- |
| `BR-411` | Bar受付済受信 | 注文状態を配送受付済へ更新 |
| `BR-412` | 配送準備中受信 | 注文状態を配送準備中へ更新 |
| `BR-413` | 配送中受信 | 注文状態を配送中へ更新 |
| `BR-414` | 配送完了受信 | 注文状態を配送完了へ更新 |
| `BR-415` | 配送失敗、持戻り、住所不備受信 | 注文状態を配送例外へ更新 |
| `BR-416` | 再配達待ち受信 | 注文状態を再配達待ちへ更新 |
| `BR-417` | キャンセル受信 | 注文状態をキャンセルへ更新 |

### 10.3 終端制御
| ルールID | ルール |
| --- | --- |
| `BR-421` | 配送完了、キャンセルは終端状態とし、後続で送信待ちや配送中へ戻さない。 |
| `BR-422` | 終端後の状態更新は、配送イベント連番と業務妥当性の両方を満たす場合のみ例外的に許容し、通常は受け付けない。 |

## 11. 対外通知ルール

### 11.1 Foo社向け配送結果返却
| ルールID | 条件 | 処理 |

### 11.2 Baz社 / Qux社向け非同期通知

| `BR-512` | Foo注文のBar向け出荷依頼送信成功 | Qux社向けに注文状態通知を行う |
| `BR-513` | 配送完了受信 | Baz社向けに確定請求通知を行う |
| `BR-514` | Foo注文の配送状態更新受信 | Qux社向けに通知キー付き状態更新通知を行う |

### 11.3 請求金額・請求日
| ルールID | ルール |
| --- | --- |
| `BR-531` | 明細税抜金額は税抜単価と数量の積とする。 |
| `BR-532` | 明細消費税額は明細税抜金額に税率を乗じ、1円未満を明細単位で切り捨てる。 |
| `BR-533` | Baz向け`tax_amount`は明細消費税額合計、`billing_amount`は明細税抜金額合計と消費税額合計の和とする。 |
| `BR-534` | 仮請求の`billing_date`はBar社受付日、確定請求の`billing_date`は配送完了イベント発生日とする。 |
| `BR-535` | 金額と税額は注文受付時のスナップショットを仮請求・確定請求の共通正本とする。 |
| `BR-536` | 算出後の税込請求金額または消費税額がBaz社の許容上限を超える注文は受付エラーとする。 |

### 11.4 状態照会 API
| ルールID | 条件 | 処理 |
| --- | --- | --- |
| `BR-521` | Foo社から状態照会 API 呼出 | 最新の注文状態、配送状態を返却する |
| `BR-522` | 同一 `partner_order_id` に複数配送イベントが存在 | 最新イベントを返却する |
