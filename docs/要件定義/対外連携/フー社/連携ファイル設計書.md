# 09. HULFTファイル設計書 Foo Order Gateway

## 1. 対象IF

- IF-ID: `IF-FOO-HOGE-001`
- IF名: Foo出荷依頼ファイル連携

## 2. 基本情報

| 項目 | 内容 |
| --- | --- |
| IF-ID | IF-FOO-HOGE-001 |
| ファイル名 | `FOO_ORDER_yyyyMMddHHmmssSSS_NNN.dat` |
| 送信元 | Foo Order Gateway |
| 送信先 | Hoge OrderHub |
| 方式 | HULFT |
| 送信タイミング | Foo社での注文確定都度 |
| 送信経路 | Foo HULFT集配信サーバ → Foo社対外接続境界 → Hoge社 Foo向け接続境界 → Hoge HULFT受信サーバ |
| 文字コード | UTF-8 |
| 改行コード | LF |
| 区切り文字 | カンマ |
| 囲み文字 | なし |
| ヘッダ行 | なし |
| HULFT受信サーバ | `hoge-hulft-rx-01` |
| ファイル着信先S3 | `s3://hoge-orderhub-landing-prod/foo/` |
| 処理済みS3 | `s3://hoge-orderhub-archive-prod/foo/processed/` |
| エラーS3 | `s3://hoge-orderhub-archive-prod/foo/error/` |
| Hoge側取込契機 | HULFT受信完了をトリガに FooOrderImportJob を即時起動 |

- `yyyyMMddHHmmssSSS` は Foo社送信時刻
- `NNN` は同一ミリ秒内の重複を避けるための送信連番
- 都度送信前提のため、同一営業日内でも複数ファイルが発生する

## 3. レコード形式

CSV形式。

```text
record_type,partner_order_id,order_type,priority_level,customer_id,item_code,quantity,unit_price_excluding_tax,tax_rate,order_datetime,delivery_zip_code,delivery_address,delivery_name,delivery_phone,package_count,payment_method,requested_delivery_date,special_instruction,shipping_release_at
```

```text
D,FO202606080001,01,0,C00000000001,ITM0000001,2,5000,10,2026-06-08T01:10:25,1000001,東京都千代田区千代田1-1,山田太郎,0312345678,1,PREPAID,2026-06-09,置き配不可,
D,FO202606080002,02,8,C00000000002,ITM0000002,1,8000,8,2026-06-08T01:12:10,1600022,東京都新宿区新宿1-1-1,鈴木花子,09012345678,1,COD,2026-06-10,,2026-06-09T08:00:00
```

## 8. 代表的なエラー例

### 8.1 対象レコード

```text
D,FO202606080999,99,0,C00000000003,ITEM-000012345,1,-100,7,2026-06-08T01:15:40,1500001,東京都渋谷区神宮前1-1-1,,ABC,0,UNKNOWN,,,
```

### 8.2 発生エラー

| 項目 | 値 | エラーコード | 理由 |
| --- | --- | --- | --- |
| order_type | 99 | E-HOGE-VAL-003 | 01, 02, 03 のいずれでもない |
| item_code | ITEM-000012345 | E-HOGE-VAL-005 | 半角英数字10桁ではない。ハイフンを含み、桁数も超過している |
| unit_price_excluding_tax | -100 | E-HOGE-VAL-014 | 1未満である |
| tax_rate | 7 | E-HOGE-VAL-015 | 許可税率ではない |
| delivery_name | 空文字 | E-HOGE-VAL-016 | 必須項目が未指定である |
| package_count | 0 | E-HOGE-VAL-018 | 1未満である |
| payment_method | UNKNOWN | E-HOGE-VAL-019 | 許可値ではない |

## 9. Hoge社での優先配送変換ルール

| 条件 | Hoge登録値 | Bar向け送信値 |
| --- | --- | --- |
| Foo注文かつ `order_type=02` かつ `priority_level>=5` | `order_source=FOO`、`partner_priority_level=5..9`、`shipping_priority_class=PRIORITY` | 優先配送電文 |
| Foo注文かつ上記以外 | `order_source=FOO`、`partner_priority_level=0..4`、`shipping_priority_class=NORMAL` | 通常配送電文 |

- Hoge社は注文受付時に受注元を `FOO` として登録する。
- Hoge社は優先度の数値自体は保持するが、Bar社向けには `PRIORITY` / `NORMAL` の二区分へ変換して送信する。
- Bar社内での配送資材配分や実際の優先制御ロジックは Bar社責務とし、Hoge社は把握しない。

## 10. 調査上のポイント

- ファイル名の接頭辞 `FOO` は `Foo Order Gateway` を表す
- ファイル着信先は `s3://hoge-orderhub-landing-prod/foo/` である
- HULFTは業務アプリへ直接到達せず、Hoge HULFT受信サーバを経由して着信S3へ配置される
- ログに出る `order_type=99` と `item_code=ITEM-000012345` は本設計書に照らして不正である

## 11. IF-HOGE-FOO-001 注文受付通知ファイル

### 11.1 基本情報

| 項目 | 内容 |
| --- | --- |
| IF-ID | IF-HOGE-FOO-001 |
| ファイル名 | `FOO_ORDER_ACK_yyyyMMddHHmmssSSS_NNN.dat` |
| 送信元 | Hoge OrderHub |
| 送信先 | Foo Order Gateway |
| 方式 | HULFT |
| 送信経路 | 注文受付通知ファイル出力S3 → Hoge HULFT送信サーバ → Hoge社 Foo向け接続境界 → Foo社対外接続境界 → Foo HULFT集配信サーバ |
| 文字コード | UTF-8 |
| 改行コード | LF |
| 区切り文字 | カンマ |
| 注文受付通知ファイル出力S3 | `s3://hoge-orderhub-status-prod/foo/ack/` |

### 11.2 レコード形式

```text
partner_order_id,receipt_status,receipt_datetime,message_code
```

### 11.3 項目定義

| 項目名 | 物理名 | 型 | 必須 | 桁数 | 説明 |
| --- | --- | --- | --- | ---: | --- |
| 提携先注文ID | partner_order_id | string | 必須 | 14 | Foo社側注文ID |
| 受付状態 | receipt_status | string | 必須 | 20 | `RECEIVED` または `RECEIVED_HOLD` |
| 受付日時 | receipt_datetime | string | 必須 | 19 | yyyy-MM-ddTHH:mm:ss |
| 通知コード | message_code | string | 必須 | 20 | 即時出荷対象は `HOGE-ACK-000`、出荷保留時は `HOGE-ACK-001` |

### 11.4 補足

- 注文ファイル取込と基本バリデーション完了後に `注文受付通知Worker` が生成する
- 本通知はBar社受付完了を意味せず、Hoge社内で注文を受け付けたことのみを示す
- Bar社向け送信が営業時間外待機となる場合でも、本通知は先に返却する
- `shipping_release_at` が未来日時の場合は `RECEIVED_HOLD` を返却し、Hoge社内では出荷解放待ちとして管理する

## 12. IF-HOGE-FOO-002 配送結果返却ファイル

### 12.1 基本情報

| 項目 | 内容 |
| --- | --- |
| IF-ID | IF-HOGE-FOO-002 |
| ファイル名 | `FOO_ORDER_STATUS_yyyyMMddHHmmssSSS_NNN.dat` |
| 送信元 | Hoge OrderHub |
| 送信先 | Foo Order Gateway |
| 方式 | HULFT |
| 送信経路 | 配送結果返却ファイル出力S3 → Hoge HULFT送信サーバ → Hoge社 Foo向け接続境界 → Foo社対外接続境界 → Foo HULFT集配信サーバ |
| 文字コード | UTF-8 |
| 改行コード | LF |
| 区切り文字 | カンマ |
| 配送結果返却ファイル出力S3 | `s3://hoge-orderhub-status-prod/foo/status/` |

### 12.2 レコード形式

```text
partner_order_id,order_status,status_datetime,delivery_company_code
```

### 12.3 項目定義

| 項目名 | 物理名 | 型 | 必須 | 桁数 | 説明 |
| --- | --- | --- | --- | ---: | --- |
| 提携先注文ID | partner_order_id | string | 必須 | 14 | Foo社側注文ID |
| 注文ステータス | order_status | string | 必須 | 20 | 配送結果に応じた注文状態 |
| ステータス更新日時 | status_datetime | string | 必須 | 19 | yyyy-MM-ddTHH:mm:ss |
| 配送会社コード | delivery_company_code | string | 必須 | 10 | `BAR` または `FUGA` |

### 12.4 補足

- 配送状態取込後に `配送結果返却Worker` が生成する
- 注文元追加は行わず、返却先は Foo社のみとする
- 配送受付済、配送準備中、配送中、配送完了などの状態変化ごとに返却対象となる
- Foo社は別途 `状態照会クライアントサーバ` から Hoge社の `出荷状態照会API` を呼び出せるが、本ファイルは正式な配送結果返却手段として継続利用する

## 13. 補足

- 物理ファイル名は `FOO_ORDER_yyyyMMddHHmmssSSS_NNN.dat` とし、都度送信でも一意になるようにする
- `shipping_release_at` 指定時、Hoge社は顧客確認・在庫引当までは即時実施するが、Bar社向け配送依頼は解放日時到来後に実行する
- 注文の部分取消は Bar社配送依頼前であれば要件候補だが、本版ではIF未採用とする
- Fuga社は配送会社であり、Hoge社とは `特殊配送依頼送信API` と `特殊配送結果通知API` で連携する
- Foo社へ返却する配送会社コードは、配送委託先に応じて `BAR` または `FUGA` を扱う
